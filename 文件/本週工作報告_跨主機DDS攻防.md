# 本週工作報告 — 跨主機 DDS 資安攻防實驗室

**期間**：2026-06-15 ~ 2026-06-16
**主題**：把資安攻防從「同機模擬」搬上「真跨主機」，並讓 Zeek 在 DDS 網路層真正偵測到攻擊
**一句話總結**：我們建好了一條攻擊機↔目標機的真實 DDS 戰場，並把 Zeek 偵測器從「只會抓偵察」升級到**五類偵測**（偵察／注入／DoS／參數竄改／來源 IP 偽造），全部用合成封包離線驗證通過、紅隊也實戰打過。

## 二、跨主機環境是怎麼建起來的（遇到的坑 & 解法）

**核心洞見**：同一台主機內的 ROS2/DDS 通訊，資料預設走**共享記憶體**，封包不會出現在網路上 → Zeek（網路層偵測）完全看不到注入/DoS。**這就是為什麼一定要跨主機。**

實際遇到的三個坑與解法：

1. **WSL2 預設 NAT，LAN 上看不到它**
   → 改用 Windows 11 的 **mirrored 網路模式**（`.wslconfig` + `wsl --shutdown`），WSL 直接拿到 LAN/直連 IP。
2. **Wi-Fi 有 AP/客戶端隔離，兩台互 ping 不到**
   → 改用**實體網路線直連**，自建獨立子網 `10.10.10.0/24`（目標機 .2、攻擊機 .1）。
3. **DDS 用 domain 30，封包跑在高埠（14900+），不是課本上的 7400**
   → 兩台 Windows 防火牆放行 inbound UDP `7400-65000`；Zeek 偵測埠範圍也同步從 7400-7500 修到 **7400-15200**（否則規則永遠不會觸發）。

**驗收**：`tcpdump` 抓到雙向 SEDP/DATA，目標機 listener 印出 `I heard: Hello World` → 跨主機 DDS 成立 ✅

## 三、Zeek 偵測器：從 1 類 → 5 類（本週核心成果）

原本 `dds_monitor.zeek` 只有「新節點（偵察）」一條規則，而且埠範圍還抓錯。本週擴充成：

| 偵測 | 攻擊手法 | Zeek 抓什麼訊號 | 狀態 |
|---|---|---|---|
| **偵察** | 未授權節點上線列舉 topic/node | 白名單外來源出現在 DDS 探索埠 | ✅ |
| **注入** | 偽造 DATA 灌進 topic | UDP payload 含注入簽章 `INJECTED` | ✅ |
| **DoS** | SPDP 探索風暴 | 8 秒內非白名單 SPDP 流量 > 25 筆 | ✅ |
| **參數竄改 (F1)** | 未授權 `set_parameters` 改節點參數 | payload 含 `set_parameters` 服務簽章 | ✅ 新增 |
| **來源 IP 偽造 (F7)** | 偽造來源 IP 成白名單繞過偵測 | **IP↔MAC 綁定**：信任 IP 配到非預期 MAC | ✅ 新增 |

**關鍵設計：信任白名單（FPR 的命脈）**
目標機自己的節點啟動時會噴大量 SPDP 探索多播，這些會出現在網路上。如果不設白名單，**Zeek 會把自己誤判成攻擊**。所以我們把目標機自己（10.10.10.2 + 其真實 MAC）列為信任來源 → 自己的正常流量不告警，只有外來的才告警。

**驗證方式**：寫了一支 `Zeek監控/test/gen_test_pcap.py`，純 Python 造出含「正常 + 五類攻擊」的合成封包，離線跑 `zeek -r`。結果：**五類攻擊全部觸發告警，合法流量零誤報**。這是可重複的單元測試，不依賴 live 環境。

## 四、紅隊滲透測試結果 & 我們的修補

紅隊（攻擊機）對目標機做了完整滲透，打出 7 個弱點：

| 編號 | 弱點 | 嚴重度 | 我們的處置 |
|---|---|---|---|
| 根因 | **SROS2 安全全關**（無驗證/加密/存取控制） | 🔴 CRITICAL | 列為下階段主修補（一招解 F1–F6）→ 後續計劃 |
| F1 | 未授權參數竄改 `set_parameters` | 🔴 CRITICAL | ✅ **已加 Zeek 偵測**（簽章比對） |
| F2 | 未授權指令注入（偽造 DATA） | 🔴 CRITICAL | ✅ 已有注入偵測 |
| F3 | 全域竊聽/資訊外洩 | 🟠 HIGH | SROS2 加密根治（後續） |
| F4 | 探索層 DoS（SPDP 灌爆） | 🟠 HIGH | ✅ 已有 DoS 偵測 |
| F5 | 明文傳輸 | 🟠 HIGH | SROS2 加密根治（後續） |
| F6 | 全系統列舉 | 🟡 MED | ✅ 偵察偵測可發現；SROS2 存取控制根治 |
| F7 | **偵測器 IP 白名單可被偽造繞過** | 🟡 MED | ✅ **已加 IP↔MAC 綁定**抓偽造 |

**最有價值的一點（F7）**：紅隊用 Scapy 偽造來源 IP 成白名單的 10.10.10.2，**把我們自己的偵測器繞過**。我們的回應是加 **IP↔MAC 綁定**：信任 IP 若配到非預期 MAC 就告警。
> **誠實說明殘留風險**：若攻擊者同時偽造 IP **和** MAC，L2 層就無法分辨 → 這正是為什麼**真正的根治是啟用 SROS2 身分驗證（每節點憑證）**，這已列入後續計劃。這不是缺陷，而是「網路層偵測的邊界」——剛好示範了為什麼需要協定層的身分驗證。
