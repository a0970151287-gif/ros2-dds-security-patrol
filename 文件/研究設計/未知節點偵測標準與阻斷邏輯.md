## 二、「惡意」的判斷標準

### 2.1 現行標準（已實作）

**判斷為惡意的症狀：節點存在性異常**

| 症狀 | 判斷方式 | 依據 |
|---|---|---|
| 出現不在白名單的節點 | 名稱比對 | monitor_node.py line 101 |
| 節點名稱不符合任何豁免前綴 | 字串前綴檢查 | monitor_node.py line 102–106 |

**弱點**：攻擊者只要把節點命名成白名單中的名稱，就能繞過偵測（身份偽造）。

---

### 2.2 依文獻補充的判斷標準

根據 6 篇 DDS Security 論文，「惡意」可以從三個層次定義：

#### 層次 A：身份驗證失敗（Authentication）
**來源**：CCS 2022 "On the (In)Security of Secure ROS2"

- **症狀**：節點加入 DDS Domain 時，無法提供有效 X.509 憑證
- **偵測方式**：SROS2 的 DDS Security Plugin 在 Authentication handshake 失敗時拋出錯誤
- **可信度**：最高——憑證是加密簽署的，攻擊者無法偽造
- **現行系統對應**：`governance.xml` 設定 `authentication_enabled=true`（已實作）

#### 層次 B：行為頻率異常（Anomaly Detection）
**來源**：ROSPaCe 2024、HCRL 2023

- **症狀**：節點發布訊息的頻率異常暴增（DoS）或 DDS Discovery 封包大量湧入（Flooding）
- **偵測方式**：監控 `/cmd_vel`、`/scan` 等 Topic 的發布頻率，超過正常值 3σ 觸發警報
- **可信度**：中——需要建立正常行為基線，有誤報風險
- **現行系統對應**：**尚未實作**

#### 層次 C：存取權限違反（Authorization）
**來源**：CCS 2022 V1–V4 漏洞分析

- **症狀**：節點嘗試 Publish/Subscribe 超出 `permissions.xml` 允許範圍的 Topic
- **偵測方式**：DDS Security Plugin 的 Access Control 層攔截並記錄
- **可信度**：高——由 DDS middleware 強制執行，不依賴應用層
- **現行系統對應**：`permissions.xml` 最小權限原則（已實作），但日誌未串接到 `/security/alerts`

## 三、具體阻斷邏輯（完整流程）

### 3.1 現行阻斷流程（已實作）

```
[偵測到未知節點]
        │
        ▼
1. 發布警報到 /security/alerts（std_msgs/String）
   ── payload 含節點名稱 + 當前總節點數
        │
        ├──► patrol_node 收到警報
        │    └── 立即設 _paused=True，停止發送 /cmd_vel
        │        30 秒後自動恢復巡邏
        │
        ├──► mission_manager 收到警報
        │    └── 任務切換為 EMERGENCY_STOP
        │        30 秒後恢復 PATROL
        │
        ├──► system_status 收到警報
        │    └── 健康報告資安欄位更新為 "🚨 警報"
        │        30 秒後清除
        │
        └──► burger_env 收到警報（RL 訓練中）
             └── _security_stop=True，本步驟 reward -50
                 下一步重置環境
        │
        ▼
2. monitor_node 直接發布零速度到 /cmd_vel
   ── 持續 3 秒（每 0.1 秒發送一次，共 30 次）
   ── 蓋過 patrol_node 的速度指令
        │
        ▼
3. 發送 LINE 推播通知操作者
   ── HTTP POST 到 LINE Messaging API（獨立執行緒，不阻塞 ROS2）
        │
        ▼
4. 取消 Nav2 導航目標（如果有在執行）
   ── /navigate_to_pose/_action/cancel_goal
```

---

### 3.2 各訂閱者反應時序

| 時間點 | 事件 |
|---|---|
| T+0s | monitor 偵測到未知節點，發布警報 |
| T+0s | patrol 停止移動（_paused=True）|
| T+0s | mission_manager 切換 EMERGENCY_STOP |
| T+0~3s | monitor 持續發送零速度到 /cmd_vel（10 Hz × 3s）|
| T+3s | monitor 停止發送零速度 |
| T+30s | patrol 自動恢復巡邏（_paused=False）|
| T+30s | mission_manager 恢復 PATROL |
| T+30s | system_status 健康報告恢復 ✅ 安全 |

## 四、現行方法的侷限與升級路徑

### 侷限

| 問題 | 說明 |
|---|---|
| 輪詢延遲 5 秒 | 攻擊者有 5 秒的視窗可以發送惡意指令再離開 |
| 只看名稱，不看憑證 | 白名單節點名稱可被偽造（GUID 不同但名稱相同）|
| 無法偵測 Topic 層攻擊 | 合法節點被入侵後，其行為異常無法偵測 |

### 升級路徑（依文獻建議）

**短期（可在現有系統上加）**：
1. 縮短輪詢間隔：5s → 1s（monitor_node 參數 `poll_interval_sec`）
2. 加入 GUID 追蹤：同名節點但 GUID 不同 → 觸發警報（身份偽造偵測）

**中期（需新增模組）**：
3. Topic 發布頻率監控：對 `/cmd_vel` 建立頻率基線，異常時警報
4. DDS Security 日誌串接：將 fastdds 的 Authentication 失敗事件導入 `/security/alerts`

**長期（RTPS 封包層，依 HCRL 2023）**：
5. RTPS 封包 Entity ID 追蹤：新 GUID 出現即時偵測（比輪詢快 5 秒）
6. Discovery 封包頻率監控：DDS Discovery flooding 偵測
