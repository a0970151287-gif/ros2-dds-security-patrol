## N21 — IDS 偵測器誘發式持續 DoS（與 N13 同類，走 IDS）

### 攻擊面圖（本輪偵察結論）
列出全系統訂閱，分「有驗章」vs「原始無驗章」：
- **有驗章**：`/security/alerts` `/security/heartbeat` `/sensor/status` `/mission/cmd` `/system/health` `/patrol/goto`
- **原始無驗章**：`/scan` `/odom` `/cmd_vel` `/imu`（訊息型別非 String，包不了簽章）

### 原理（白話）
IDS（入侵偵測）只要一偵測到異常，就**自己用密鑰簽一個真警報**廣播 → patrol 收到就停車 30 秒。
攻擊者沒密鑰，但只要對 `/cmd_vel` 發一筆訊息（用一個「不在白名單」的節點名），
IDS 的 D4 偵測器就會說「有未授權的 cmd_vel publisher！」→ **替攻擊者簽警報**。

**關鍵時間差（這是真正可修的 bug）**：
```
IDS 警報冷卻 ALERT_COOLDOWN_SEC = 10 秒（兩次警報最少間隔）
patrol 恢復計時器 resume timer  = 30 秒（停車後多久恢復）
        10s  <  30s
```
因為 IDS 自己每 10 秒就能再發一次警報，而 patrol 要 30 秒才恢復——
**patrol 的 30 秒倒數永遠被新警報重置 → 永久停車。**
攻擊者甚至不必很勤勞，每 8~10 秒戳一下就夠。

### 攻擊鏈
```
攻擊者(無密鑰)                IDS(有密鑰)                  patrol
─────────────────────────────────────────────────────────────────
用名字 evil_cmd_driver
發一筆 /cmd_vel  ──────────▶  D4: 未授權 publisher!
                              簽 CH_ALERTS 警報  ────────▶  驗章通過(章是真的)
                                                            🚨 停車, 啟動 30s 倒數
(每 8s 重複)  ─────────────▶  (每 10s 簽一次)  ──────────▶  倒數一直被重置 → 永久停
```

### 整合測試證據（live）
- 環境：IDS + patrol（域 99）
- 攻擊：`evil_cmd_driver` 每 8s 發一筆 `/cmd_vel`（值 0.10，正常範圍，**不需極端值**）

| 指標 | 數值 |
|---|---|
| 攻擊者戳 /cmd_vel | 6 次 |
| IDS 簽警報 | 每 ~10s 一次（537.44 → 547.94 → …）|
| patrol 停車 | 1 次 |
| **patrol 恢復** | **0 次（45 秒全程停擺）** |

鐵證——patrol 停車理由：
```
🚨 安全警報（已驗章）！巡航停止: 🛡️ [智能防禦警報]
                                     ↑
            patrol 服從的是「IDS 因攻擊者的未授權 cmd_vel 而簽出的真警報」
```
攻擊者無密鑰，卻讓 patrol 永久停車。

### 嚴重度與定位（誠實）
- **這是真攻擊**：未授權 L1 攻擊者用一筆正常 cmd_vel + 重複，就能讓機器人**永久停止**（阻斷生產）。
- **一半屬於已揭露範圍**：發 `/cmd_vel` 本身是 raw-topic 攻擊面（藍方已標「需 SROS2 根治」）。
- **但有一個獨立於 SROS2、應用層就能修的真 bug**：
  **IDS 冷卻 10s < patrol 恢復 30s 的時間差**——即使沒有 SROS2，把這個時間差修正，
  單一觸發源就無法造成「永久」停車（最多間歇）。

### 修補建議
1. **時間差**：讓 patrol 的 resume timer ≤ IDS cooldown，或「重複同源警報不無限重置倒數」
   （例如同一 detector 訊號在短時間內只算一次）。
2. **與 N13 同根的結構修法**：偵測器偵測到異常**不該無條件 cascade 成 emergency stop**；
   對「來自未驗證 raw topic 的觸發」應 log + 限流 + 人工研判，而非自動全系統停車。
3. **根治 cmd_vel 注入**：SROS2 Enforce（攻擊者根本發不了 /cmd_vel）。

## N22 — `/imu` 無驗章且不在 D4 監控（低）

### 確認
- `sensor_hub_node` 訂閱 `/imu`（[sensor_hub_node.py:37](../src/dds_security_monitor/dds_security_monitor/sensor_hub_node.py#L37)），算出水平加速度，放進 `/sensor/status`。
- IDS 的 D4 publisher 偵測**只查 `/cmd_vel` `/scan` `/odom`**（[intelligent_defense_node.py:222-224](../src/dds_security_monitor/dds_security_monitor/intelligent_defense_node.py#L222-L224)）——**`/imu` 不在內**。
- `/imu` 訊息無簽章。

→ 攻擊者可自由偽造 `/imu`，且**不會被任何偵測器抓到**。

### 為何只列低
sensor_hub 把假加速度簽進 `/sensor/status`（簽章合法 → 下游相信），
但下游的任務決策是看 `min_range`（來自 /scan）判斷「危險/安全」，**不看加速度**。
所以偽造 /imu 目前只會讓**儀表板顯示假的加速度數字**，不影響控制決策。

### 但帶出一個架構觀察（值得寫進報告）
這是 **「signed garbage（簽了章的垃圾）」** 問題的縮影：
N6 修補讓 sensor_hub 對 `/sensor/status` 蓋章，但蓋章只證明「**這是 sensor_hub 發的**」，
**不證明「sensor_hub 依據的原始輸入（/scan、/imu）是真的」**。
若未來有任何安全決策開始依賴 /imu 或更深的感測鏈，這個「輸入未驗證 → 輸出被簽章背書」
的縫就會被放大（垃圾進、蓋章出）。

### 修補建議
- 短期：把 `/imu` 也納入 D4 的 publisher 計數/名稱檢查（與 scan/odom 一致）。
- 根本：raw 感測 topic 的真實性靠 SROS2；應用層的「簽章」不要被誤解為「資料正確性背書」。

## N23 — 變體驗證：白名單合法名字一樣能觸發（證明 N21 是結構問題）

### 目的
證明 N21 不是「D4 名字檢查」單一 detector 的洞——就算攻擊者用**白名單上的合法名字**也擋不住。

### 實測（誠實，含我假設被推翻的部分）
- 攻擊者 node 名 = `teleop_keyboard`（在 CMD_ALLOWED 白名單）
- 振盪 cmd_vel ±0.5（想觸發 D1 物理 + D2 方向衝突，**避開 D4**）

**結果（精確）**：4 筆警報全是 `vote=2/6`，detector 分解：
```
• D1[cmd_lin=0.50>=0.23]                                    ← 行為偵測(物理)
• D4[cmd 同時 2 個 publisher: ['teleop_keyboard','patrol_node']] ← 計數檢查(非名字)
```
patrol：**停車 1 次、恢復 0 次（永久停）**。

### 誠實修正 + 更強的結論
我原本假設「純 D1+D2、不踩 D4」——**這點錯了**：D4 還是觸發了，但走的是**計數路徑**
（teleop_keyboard + 真 patrol = 2 個 cmd_vel publisher），**不是名字路徑**（名字是合法的）。

但這讓結論**更強**：
1. **名字白名單沒救** — 名字是合法的 `teleop_keyboard`，D4 名字檢查放行了，攻擊照樣成功。
2. **D1 行為偵測與名字無關** — 不管你叫什麼，速度超標就觸發。
3. **D4 計數也與名字無關** — 只要攻擊者跟真 patrol 同時發 cmd_vel，就是「2 個 publisher」。

→ **修 D4 的名字檢查補不了 N21。** 這是「偵測→簽警報→30s 停車」cascade +
「IDS 冷卻 10s < patrol 恢復 30s」時間差的**結構問題**。唯一的應用層解是改 cascade/時間差（見 N21 修補建議）。

### N21 的意義（報告金句）
> **「偵測到攻擊」不該等於「自動全系統停車」。**
> 當「觸發偵測」這件事可以被未授權者輕易做到（發一筆 raw topic），
> 而「偵測的後果」是會 cascade 的緊急停止時——**偵測器本身就變成攻擊者的遙控停車按鈕**。
> 這跟 N13 是同一個教訓：**防禦動作的「觸發門檻」必須跟「觸發者的可信度」匹配。**
