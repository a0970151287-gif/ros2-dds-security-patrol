## 🔴 紅隊成功攻擊（N1–N7, N13）— 8 個，當時都有 log 證據

### N1 — 心跳重放（Heartbeat Replay）✅
- **白話**：警衛每 2 秒喊「我沒事」，這句話有蓋章但沒寫時間。錄起來一直重播，沒人發現警衛倒了。
- **技術**：monitor 每 2s 發 HMAC 簽章心跳給 IDS；簽章 envelope 無 timestamp/nonce → 側錄 1 筆無限重放。
- **證據**：IDS 心跳看門狗(D5)應 10s 內 fire，實測 **0 次**；攻擊者重放 30 次 IDS 統計全 0。
- **修補**：envelope 加 ts + nonce + 接收端 ReplayCache 記憶已見過的 nonce。

### N2 — 偽裝系統內部節點（Regex 後門）✅
- **白話**：警衛靠「名字長得像員工」放行。取名「員工-清潔-007」就大搖大擺走進去。
- **技術**：monitor 白名單用正則放行 `_ros2cli_*` 等「ROS 內部節點」；命名 `_ros2cli_evil_99999_deadbeefcafe` 符合格式 → 不報警。
- **證據**：monitor 全程 0 報警；攻擊者存活 8s 自報「monitor 看不到我」。
- **修補**：移除「用名字格式判斷」的白名單，改「開機快照 + 只信白名單」。

### N3 — 警報重放 → 永久停車（Alert Replay DoS）✅
- **白話**：警報一響機器人停 30 秒。把警報錄起來每 5 秒按一次 → 永遠停。不需密鑰。
- **技術**：警報 HMAC envelope 同樣無 timestamp → 側錄合法警報，每 5s 重放（< 30s resume timer）→ patrol 永久 paused。
- **證據**：35s 內 patrol **0 次 resume**，重放 8 次。
- **修補**：警報 envelope 也加 ts + nonce。

### N4 — 跨頻道蓋章混淆（Cross-Channel Confusion）✅
- **白話**：「我沒事」公文和「緊急警報」公文用同一顆章、又沒寫用途。把警衛剛蓋的「我沒事」丟進警報信箱就觸發停車。
- **技術**：N3 修補後加了 ts/nonce，但 heartbeat 與 alerts 共用同一 sign 格式、同密鑰、envelope 無 channel 欄位。心跳每 2s 一個新鮮 nonce → 轉貼到 `/security/alerts` → patrol 驗章通過(章真、時間新、nonce 沒見過) → 停車。
- **證據**：patrol 停車理由印 `hb|1779976511.662`（心跳格式，非警報文字）。
- **修補**：envelope 加 `channel` 欄位，驗章比對 `expected_channel`；轉貼的心跳因 `channel: heartbeat ≠ alerts` 被拒。**紅隊回測已確認擋下。**

### N5 — 開機前卡位汙染基準（Pre-startup Baseline Poison）✅
- **白話**：警衛上班會把「現在在場的人」記成自己人。比警衛早到躲進去 → 被當老員工。
- **技術**：monitor 首次 poll 把當下整個 graph 吸收進 baseline。攻擊者比 monitor 早啟動 → 進 baseline → 永不報警。
- **證據**：monitor 印「基準快照 3 個節點」(含 `totally_evil_node`)，21s 全程沉默。
- **修補**：首次 baseline 改成「只把白名單上的當基準」，非白名單一律報警。

### N6 — 偽造感測器狀態（/sensor/status Spoof）✅
- **白話**：內部便條不蓋章、只寫「來自：感測部」。冒名寫便條「四周很安全」，任務部全信。
- **技術**：`/sensor/status` 訊息原本完全無簽章；攻擊者偽裝白名單名 `sensor_hub_node`(monitor 不報警) + 高頻發假狀態 → mission_manager 照單全收。
- **證據**：mission_manager 跟著切 7 輪「避障/巡邏」。
- **修補**：sensor_hub 改用簽章發送(綁 CH_SENSOR)，下游驗章。

### N7 — 偽造任務指令 → 騙操作員儀表板（/mission/cmd Spoof）✅
- **白話**：同 N6 但打更下游。冒名「任務部」直接竄改老闆看的儀表板。
- **技術**：`/mission/cmd` 無簽章；偽裝 `mission_manager_node` 灌假任務 → `/system/health` 顯示攻擊者字串。
- **證據**：/system/health 印 `ATTACKER_OWNED_MISSION_STATE`。
- **修補**：mission_manager 改簽章發送(綁 CH_MISSION)，system_status 驗章。

### N13 — 自我監控反射放大 + 永久致盲 ✅（最狡猾）
- **白話**：監控室新規「看到假健康報告就自己拉警報」。我塞鬼畫符 → 監控室「這假的!」→ 自己拿章拉真警報停車（借刀殺人）。而且這規則只觸發一次就永久關閉 → 我先塞一張，這條防線從此瞎掉。
- **技術**：N8 self-watch 對「未簽章垃圾」也反應 → system_status 用 secret 簽 CH_ALERTS 廣播 → patrol 停車（confused deputy）；`_health_impostor_alerted` latch 永不 reset → 一封包永久致盲。
- **證據**：patrol 停車理由印 `🚨 安全警報（已驗章）！...🛑 [N8 防護]`；戳 5 次只反射 1 次。
- **修補**：未簽章垃圾只 log 不反射、移除 latch 改 rate-limit。**紅隊回測已確認擋下。**

## 🔵 藍方主動修補（N8–N12, N14, N15, N17）— 守方自己預想的攻擊

> 這些不是紅隊打的，是藍方「我如果是駭客會怎麼打」自問自答 + 搶先補。報告可寫成「防守方主動威脅建模」。

### N8 — 儀表板偽造 + 自我監控 🔵
- **藍方預想**：攻擊者偽造 `/system/health` 騙操作員。
- **藍方修補**：system_status 簽 CH_HEALTH + 自我監控(訂自己發的、看到 impostor 就示警)。
- **後續**：這個自我監控反而被紅隊 **N13** 反打（見上）。教訓：新防禦也要做威脅建模。

### N9 — 緊急停止時搶 cmd_vel（race）🔵
- **藍方預想**：緊急停止瞬間，攻擊者用高頻 cmd_vel 蓋過煞車指令。
- **藍方修補**：D1 物理門檻收緊 0.25→0.23；patrol 暫停時把送 0 速度的頻率拉到 100Hz 跟攻擊者競爭（最後一筆贏）。
- **狀態**：緩解約 62%（100Hz 攻擊下機器人 62% 時間是停的）；**根治需 SROS2**。

### N10 — 雷射掃描漂移污染 🔵
- **藍方預想**：攻擊者餵假 `/scan`（漸變而非固定，繞過「連續幀相同」偵測 D3）。
- **藍方修補**：新增 **D6 行為一致性偵測**（里程計顯示在移動，但掃描完全靜止 = 矛盾 = 攻擊）。
- **狀態**：部分擋；`/scan` 是非 String 訊息包不了簽章，**根治需 SROS2**。

### N11 — 里程計位置污染 🔵
- **藍方預想**：攻擊者餵假 `/odom`，劫持機器人對「自己在哪」的認知。
- **藍方修補**：D4 publisher 偵測延伸到 `/odom`（雙來源 = 攻擊）+ D6（指令在推進但里程計不動 = 矛盾）。
- **狀態**：部分擋；**根治需 SROS2**。

### N12 — 重放快取洪水擠出（LRU flood）🔵
- **藍方預想**：ReplayCache 用 LRU，攻擊者灌大量唯一 nonce 把「還新鮮的舊 nonce」擠出快取 → 之後可重放那個被擠出的。
- **藍方修補**：ReplayCache 改 TTL 過期（與新鮮度窗對齊），過期才清；還在新鮮窗內的 nonce 絕不被擠掉。
- **狀態**：已補。

### N14 — 參數服務劫持白名單 🔵
- **藍方預想**：每個節點都開 `/set_parameters` 服務，攻擊者把自己名字加進 monitor 白名單 → 永久隱身。
- **藍方修補**：安全相關參數全設 `read_only=True`，改不了。
- **狀態**：已補（monitor 完整；紅隊指出 **patrol 的參數沒鎖**，但目前 read-once 無實效，低危）。

### N15 — log 洗版（log storm）🔵
- **藍方預想**：攻擊者狂丟垃圾，每個接收端對每筆印一行「已拒絕」→ log 爆掉/洗版。
- **藍方修補**：拒絕訊息的 log 加 throttle（限頻）。
- **狀態**：已補。（紅隊 **N20** 後來測試「沒 throttle 的是驗章」這個縫，見下。）

### N17 — 通知洪水稀釋 🔵
- **藍方預想**：攻擊者用 100 個唯一名字灌 monitor → LINE 被 100 條通知洗版。
- **藍方修補**：LINE 通知改「每 30s 聚合成 1 條」。
- **狀態**：已補；紅隊指出殘留——預覽只列前 10 個名字（真兇可被埋）、通知延遲最多 30s（低危）。

## 🔴 紅隊後期攻擊（N18, N19, N20）— 1 個灰色 + 2 個失敗

### N18 — 監控清單無上限成長 → 記憶體耗盡 ⚠️（已證實，未完成 live）
- **白話**：monitor 把「報過警的名字」記在一張只增不減的清單上。攻擊者不斷換新名字 → 清單無限長 → monitor 記憶體爆掉。
- **技術**：`_alerted_nodes` set 在預設設定(`alert_on_node_exit=False`)下永不移除元素。
- **證據（量化）**：用真實 monitor 程式碼 churn 10,000 個唯一名字 → 清單線性長到 10,000、零縮減（~211 bytes/個）。
- **誠實標註**：**我沒有真的把運行中的 monitor 撐爆 OOM**，所以是「已證實漏洞」非「完成攻擊」。
- **修補**：藍方改 TTL(10 分鐘過期) + 上限 2048 筆 → 已 bounded（~400KB 封頂）。

### N19 — 參數服務洪水（Parameter-service Flood）❌ 失敗
- **白話**：每個節點都開著「查詢參數」的服務窗口，關不掉。狂打那個窗口想癱瘓節點。
- **技術**：高併發呼叫 `/dds_security_monitor/get_parameters`，想餓死單執行緒 executor。
- **實測**：**1,626 req/s**，monitor 心跳 **0.5Hz 紋風不動**（std dev 0.0001s）。
- **為何失敗**：唯讀參數的回應太便宜，餓不死計時器；rclpy 排程公平交錯。

### N20 — 驗章洪水（Verify-flood）❌ 失敗（鎖定 N15 的縫）
- **白話**：N15 只限制了「印 log」，沒限制「驗章」。我狂灌垃圾，逼每個節點對每筆都算一次密碼學驗章，想把 CPU 燒光。
- **技術**：每筆 `/security/*` 訊息都觸發 `verify_alert`(JSON parse + HMAC-SHA256)，不論拒不拒。
- **實測 A（BEST_EFFORT 心跳）**：**368,859 msg/s** → IDS 計時器 5.000s **不動**。失敗原因：BEST_EFFORT QoS 在洪水下**自動丟包**，IDS 只撈到極少數。
- **實測 B（RELIABLE 警報）**：**18,213 msg/s** → system_status 計時器 2.000s **不動**。失敗原因：**KEEP_LAST 深度只留 10 筆** + 排程公平，每輪驗章量被鎖死。
- **意義**：N15 看似留縫(沒 throttle 驗章)，但被更底層的 QoS 補上 = **縱深防禦**。洪水類 DoS 不需額外防禦。
