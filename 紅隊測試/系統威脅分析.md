# 系統威脅分析（System Threat Analysis）

> 文件目的：依教授指導「**情境是系統，不是人**」重寫。本文件從**軟體模組對軟體模組**的角度描述本系統的攻擊面、威脅與緩解。
>
> 寫作原則：
> - **不出現「人」**（不寫「攻擊者」「操作員」「員工」「警衛」「工程師」「使用者」當作攻擊或受害角色）。
> - **主體只有軟體模組**。攻擊方是一支「未授權程式（rogue process）」；受害方是某個被命名的軟體模組（例：`monitor_node`、`patrol_node`）。
> - **攻擊一律透過介面發生**。介面 = DDS topic / DDS service / Linux 檔案 / Linux IPC。每個介面都是攻擊面。
> - **威脅分析從介面切入，威脅評估用「高 / 中 / 低」**。
>
> 對照舊文件：
> - `THREAT_MODEL.md` — CIA / 攻擊者能力等級表，本文件不重複。
> - `ARCHITECTURE.md` — 系統拓樸圖。本文件 §1 以縮圖呈現，§2 補上對外介面清單。
> - `N1-N20_完整目錄.md` — 攻擊清單。本文件 §4 用軟體模組敘事重述代表性攻擊。

## 1. 系統情境（System Context）

本系統由 **11 個 ROS2 軟體模組** 組成，跑在單一 ROS2 DOMAIN（`ROS_DOMAIN_ID=30`）內。模組之間透過 DDS 中介軟體（Cyclone DDS）以「topic 發布 / 訂閱」「service 呼叫」傳遞訊息。

## 3. 介面威脅分析（Per-Interface Threat Assessment）

每個介面對應一個威脅，威脅評估等級 = **高 / 中 / 低**。等級依「攻擊成本 × 攻擊後果」評估：

- **高**：成本低（單一 ROS2 程式啟動即可）+ 後果嚴重（控制權失守 / 模型污染）
- **中**：成本中（需共存於同 DDS domain + 額外條件）+ 後果可逆 / 範圍受限
- **低**：成本高（需密鑰 / root）或後果不直接危害任務

| # | 介面 | 威脅敘述（軟體角度） | 風險 | 對應緩解 |
|---|---|---|---|---|
| T-01 | `/cmd_vel` | 未授權程式持續發布 Twist → 模擬器執行該指令（DDS 預設 last-writer-wins，最後一筆贏） | **高** | `intelligent_defense_node` D1 物理門檻 + D4 publisher 計數偵測 |
| T-02 | `/scan` | 未授權程式持續發布 LaserScan → `burger_sac_env` 取到偽資料污染策略；`sensor_hub_node` 把偽資料聚合進 `/sensor/status` | **高** | D3 frame-repeat + std<0.01 + 95% near-max 異常檢查；D6 odom-vs-scan 一致性 |
| T-03 | `/odom` | 未授權程式發布偽 Odometry → `patrol_node` 位置認知漂移、無法到達 waypoint | **中** | D4 雙 publisher 偵測；D6 cmd-vs-odom 一致性 |
| T-04 | `/security/alerts` | 未授權程式發布字串 → 全部下游模組進入 emergency stop | **高** | HMAC envelope v3 簽章 + ts + nonce + channel；ReplayCache 擋重放 |
| T-05 | `/security/heartbeat` | 未授權程式錄製合法心跳並重放 → `intelligent_defense_node` 看門狗永遠不超時，誤判 `monitor_node` 仍在運作 | **高** | envelope v3（ts + nonce）+ 接收端 ReplayCache；RELIABLE + TRANSIENT_LOCAL QoS |
| T-06 | `/patrol/goto` | 未授權程式發布座標 → `patrol_node` 把機器人開到任意位置 | **高** | HMAC 驗章 + 座標 ±2.5m 範圍檢查 |
| T-07 | `/sensor/status` | 未授權程式偽冒 `sensor_hub_node` 名稱 → `mission_manager_node` 切到錯誤任務模式 | **中** | 訊息綁 channel=CH_SENSOR；下游 `expected_channel` 驗章 |
| T-08 | `/mission/cmd` | 未授權程式偽冒 `mission_manager_node` 名稱 → `system_status_node` 健康狀態被竄改 | **中** | 訊息綁 channel=CH_MISSION；下游 `expected_channel` 驗章 |
| T-09 | `/system/health` | 未授權程式偽冒 `system_status_node` 名稱 → self-watch 反射觸發 emergency alert（confused deputy） | **中** | self-watch 改為 log-only + rate-limit cooldown；移除 latch |
| T-10 | `/patrol/reload` | 未授權程式高頻呼叫 service → `patrol_node` 控制 loop 卡住 | **中** | 5 秒 rate-limit；參數 `enable_reload_service` 預設停用 |
| T-11 | `/<node>/set_parameters` | 未授權程式呼叫 set_parameters → 改 `monitor_node` 白名單為 `["*"]` → 永久隱身；改 `intelligent_defense_node` 門檻 → 關閉偵測 | **高** | 所有敏感參數 `ParameterDescriptor(read_only=True)`；service 自動拒絕 |
| T-12 | ROS graph | 未授權程式採用「白名單前綴」（如底線開頭）→ `monitor_node` 將其視為內部模組放行 | **中** | 移除 prefix 白名單；改用「baseline 快照 + grace period 只吸收白名單模組」 |
| T-13 | `~/.config/dds-monitor/*` | 同 user 程式讀檔 → HMAC 密鑰、LINE token 外洩 | **低** *(陳述為 out-of-scope；超出本系統能擋的範圍)* | chmod 600；fingerprint 開機印出便偵測一致性 |
| T-14 | `runs_top/models/*.pkl` | 任意程式覆寫檔案 → `burger_sac_env` load 時 `pickle.load()` 觸發 RCE | **高** | `.sha256.hmac` 檔案簽章；load 前驗章；驗證失敗 `sys.exit(2)` |
| T-15 | `runs_top/models/*.zip` | 任意程式覆寫模型權重 → `burger_sac_env` 部署被注入策略 | **高** | 同 T-14 |
| T-16 | `/proc/<pid>/environ` | 同 user 程式讀 `/proc` → 環境變數中的 token 外洩 | **低** | 從 `credentials` 移除 export，改為檔案讀取 |
| T-17 | LINE 推播管線（外部 HTTPS） | 未授權程式持續觸發 alert → `monitor_node` 重複推播洪水 | **中** | 30 秒 batch 聚合；leading-edge 立即送；pending list 上限 256 |
| T-18 | IDS 偵測 → 自動 emergency stop（**結構性介面**） | 未授權程式故意被偵測 → IDS 觸發 stop alert → 模組鏈進入永久暫停（cascade DoS） | **中** | resume timer 不再 reset on alert；90 秒內 ≥2 次 pause → 120 秒 quiet window 等待外部介入 |

## 4. 攻擊情境（Attack Scenarios — 軟體模組敘事）

以下情境**完全不引入「人」**。每個情境的主體都是命名的軟體模組或一支未授權程式。

### 情境 1：心跳重放 → 看門狗失效（T-05）

#### 介面機制
`monitor_node` 與 `intelligent_defense_node` 之間透過 DDS topic `/security/heartbeat` 通訊。`monitor_node` 每 2 秒發布一筆心跳訊息，欄位包含 HMAC 簽章。`intelligent_defense_node` 訂閱此 topic，若 10 秒內未收到任何心跳，視為 `monitor_node` 已不存在，觸發 watchdog 警報。

此機制屬於**系統對系統的存活檢測（liveness probe）**：模組 A 定期送出短訊息，模組 B 用「最近一次收到的時間戳」判斷模組 A 是否仍在運作。

#### 介面弱點
舊版 HMAC envelope 僅包含 `{payload, sig}`，不含 timestamp 與 nonce。同一筆合法訊息送出後永遠通過驗章。

#### 攻擊軟體模組
一支以 ROS2 Python API 啟動的未授權程式（命名為 `rogue_heartbeat_publisher`），透過 `ros2 run` 或腳本啟動，加入相同 `ROS_DOMAIN_ID`。

#### 攻擊機制
1. `rogue_heartbeat_publisher` 訂閱 `/security/heartbeat`，側錄一筆合法心跳（含完整簽章）。
2. `rogue_heartbeat_publisher` 以 5 Hz 持續發布該筆訊息至 `/security/heartbeat`。
3. 另一支或同一支未授權程式對 `monitor_node` 發動拒絕服務（佔用 CPU、刪除其 `/proc/<pid>`、或觸發 OOM），使 `monitor_node` 軟體模組停止運作。
4. `intelligent_defense_node` 持續收到「看似新鮮」的心跳，watchdog 計時器永不超時。

#### 受害模組鏈
- `intelligent_defense_node`：watchdog 失效；不知 `monitor_node` 已死。
- `patrol_node` / `burger_sac_env`：依賴 `/security/alerts` 廣播緊急狀態；上游偵測層已死，alerts 不再發出。
- `system_status_node`：聚合健康狀態時把 `monitor_node` 列為 GREEN，整體系統狀態回報為 OK。

#### 風險評估
**高**。攻擊成本 = 一支 ROS2 Python 程式（無需密鑰）；後果 = 偵測層全鏈失效，後續任何攻擊不會被告警。

#### 緩解（已實作）
- HMAC envelope v3 加入 `ts`（unix epoch 秒）、`nonce`（16 bytes random）。
- `intelligent_defense_node` 帶 `max_age=3.0`：訊息 `ts` 超過 3 秒一律拒絕。
- 接收端 `ReplayCache`：記錄已見 `nonce`，重複出現一律拒絕；快取以 TTL eviction 對齊 `max_age`。
- 訊息綁 `channel=CH_HEARTBEAT`；用於 `/security/heartbeat` 的簽章不可轉用至其他 topic（擋 T-04 cross-channel）。
- `/security/heartbeat` QoS 改為 RELIABLE + TRANSIENT_LOCAL，使「DDS 預設 BEST_EFFORT 攻擊冒充 publisher」失敗。

### 情境 2：警報重放 → 控制模組永久停車（T-04）

#### 介面機制
`monitor_node` 與 `intelligent_defense_node` 透過 `/security/alerts` 廣播緊急狀態。`patrol_node`、`burger_sac_env`、`mission_manager_node`、`system_status_node` 訂閱該 topic。任一訂閱模組收到合法簽章的緊急訊息後，啟動「30 秒 resume timer」進入暫停狀態。

#### 介面弱點
舊版 alerts envelope 不含 ts/nonce；單筆訊息可無限重放。

#### 攻擊軟體模組
一支未授權程式 `rogue_alert_replayer`。

#### 攻擊機制
1. `rogue_alert_replayer` 訂閱 `/security/alerts` 等待一筆合法警報。
2. 每 5 秒（< 30 秒 resume window）重發該筆訊息。
3. `patrol_node` 的 resume timer 每次都被 reset 在「即將恢復」前。

#### 受害模組鏈
- `patrol_node`：`_cmd_pub` 持續送 0 速度，機器人不前進。
- `burger_sac_env`：訓練 episode 提早結束（環境判定為 paused），訓練步數累積但 reward 為 0。
- `mission_manager_node`：任務模式卡在「avoidance」，無法切回 patrol。

#### 風險評估
**高**。

#### 緩解（已實作）
- alerts envelope 同樣加 ts + nonce + channel=CH_ALERTS。
- 各訂閱模組各自帶 `ReplayCache`。
- T-18（cascade DoS）補強：`patrol_node` 的 resume timer 不再 reset on alert（首次 pause 30 秒之後固定到期）；90 秒內 ≥2 次 pause → 進入 120 秒 quiet window，不再自動 pause。

---

### 情境 3：節點偵測模組被「前綴白名單」繞過（T-12）

#### 介面機制
`monitor_node` 定期（5 秒）poll ROS graph 取得當前活躍的 node 名稱清單，比對白名單。不在白名單者 → `/security/alerts` 發出警報 + LINE 推播。

#### 介面弱點
舊版白名單使用 prefix 比對。任何名字以 `_` 開頭的 node 視為 ROS2 內部 daemon（如 `_ros2cli_daemon_*`）放行。

#### 攻擊軟體模組
任意一支 ROS2 程式，啟動時改名為以底線開頭（`ros2 run pkg node --ros-args --remap __node:=_attacker_disguised`）。

#### 攻擊機制
1. 未授權程式啟動，註冊到 DDS discovery，名稱為 `_attacker_disguised`。
2. `monitor_node` 下一輪 poll 看到該 node，比對白名單 prefix `_` → 視為內部 daemon → 放行。
3. 未授權程式長期駐留，可繼續對其他介面（T-01、T-02、T-04…）發動下游攻擊。

#### 受害模組鏈
- `monitor_node`：偵測層失效。
- 整個下游模組鏈失去「未授權模組存在」這個事件訊號。

#### 風險評估
**中**（攻擊本身不直接造成損害，但讓所有後續 topic-layer 攻擊變得不可見）。

#### 緩解（已實作）
- 移除 prefix 白名單。
- 改為「開機快照 baseline + grace period 內只吸收白名單模組」。
- 非白名單模組永遠視為陌生模組 → 警報。

---

### 情境 4：跨頻道簽章混淆（T-04 子情境）

#### 介面機制
`/security/heartbeat` 與 `/security/alerts` 屬不同 topic，但訊息物件結構與簽章函式相同。

#### 介面弱點（T-05 修補後新發現）
N3 修補後 envelope 加了 ts/nonce，但**簽章不綁定 topic 名稱**。一筆心跳訊息（含合法簽章 + 新鮮 ts + 未見 nonce）丟到 `/security/alerts` 訂閱端驗章會通過 — 因為訂閱端只驗「簽章是否合法、訊息是否新鮮」。

#### 攻擊軟體模組
`rogue_channel_confuser`。

#### 攻擊機制
1. `rogue_channel_confuser` 訂閱 `/security/heartbeat`，攔截最新一筆心跳。
2. 將該筆訊息直接 publish 至 `/security/alerts`。
3. `patrol_node` 訂閱端驗章通過 → 觸發 emergency stop。

#### 受害模組鏈
- `patrol_node`：根據訊息 payload 印出停車理由為「hb|1779976511.662」（心跳格式），不是 alert 格式。
- 觀察 log 才會發現訊息來源為心跳 — 一般情況下停車事實已成立。

#### 風險評估
**高**。

#### 緩解（已實作）
- envelope 加入 `channel` 欄位（值：`CH_ALERTS` / `CH_HEARTBEAT` / `CH_SENSOR` / `CH_MISSION` / `CH_HEALTH` / `CH_GOTO`）。
- 接收端呼叫 `verify_alert(..., expected_channel=CH_ALERTS)`；channel 不符立刻拒絕。

---

### 情境 5：偵測器本身被當作攻擊放大器（T-18）

#### 介面機制（結構性 — 不是單一 topic）
`intelligent_defense_node` 的設計：偵測到 D1/D2/D4 任一條件成立 → 自動發 `/security/alerts` → 下游模組進入 30 秒暫停。

此設計本意是「最後一道防線」，但**「偵測 → 自動 alert → 自動暫停」這個結構本身**就是一個介面：任何能觸發偵測條件的程式，等於擁有了觸發 emergency stop 的能力。

#### 介面弱點
偵測器是「為了攻擊者方便」設計的觸發按鈕。

#### 攻擊軟體模組
`rogue_detector_baiter`。

#### 攻擊機制
1. `rogue_detector_baiter` 用未授權名稱發布 `/cmd_vel`（觸發 D4 unauthorized publisher）。
2. `intelligent_defense_node` 偵測 → 簽章發 `/security/alerts`。
3. `patrol_node` 進入 30 秒 pause。
4. 在 30 秒 resume window 結束前再次重複步驟 1，並重置 resume timer。
5. `patrol_node` 永遠卡 pause。

注意：本攻擊**不需偽造簽章**。它利用合法的偵測 → 警報鏈。

#### 受害模組鏈
- `patrol_node`、`burger_sac_env`：cascade DoS。
- `intelligent_defense_node`：本身被當作攻擊鏈中繼。

#### 風險評估
**中**。攻擊鏈長但無門檻。

#### 緩解（已實作）
- `patrol_node` resume timer **不再 reset on alert**（首次 pause 後 30 秒固定到期）。
- `patrol_node` 加 cascade-DoS 偵測：90 秒內 ≥2 次 pause → 進入 120 秒 quiet window，不再因 alert 自動 pause。
- 設計教訓記入 `lesson_detector_as_attack_surface.md`：**每個新偵測器都是新攻擊面，上線前必做威脅建模**。

---

### 情境 6：模型檔案被覆寫（T-14、T-15）

#### 介面機制
`burger_sac_env` 啟動部署時呼叫 `model.load("tqc_best.zip")` 與 `model.load_replay_buffer("tqc_buffer.pkl")`。後者內部呼叫 `pickle.load()`。

#### 介面弱點
`pickle.load` 在反序列化時會呼叫物件的 `__reduce__()` 方法。若物件回傳 `(os.system, ('rm -rf /', ))`，反序列化過程會執行該指令。

#### 攻擊軟體模組
任意能寫入 `runs_top/models/` 的程式 — 可能是經由其他漏洞拿到相同 user 寫入權限的程式。

#### 攻擊機制
1. 未授權程式生成一個含惡意 `__reduce__` 的 pickle 檔案。
2. 覆寫 `tqc_buffer.pkl`。
3. `burger_sac_env` 下次啟動時 `load_replay_buffer` → 觸發 RCE。

#### 受害模組鏈
- `burger_sac_env`：RCE，process 被替換。
- 對 `runs_top/`、`~/.config/dds-monitor/` 有讀寫權的所有其他資源都暴露。

#### 風險評估
**高**。

#### 緩解（已實作）
- 寫檔同步寫 `.sha256.hmac`（HMAC-SHA256 over 整個檔案 bytes）。
- `burger_sac_env` load 前先呼叫 `verify_file()`；失敗 `sys.exit(2)`。
- 簽章密鑰位於 `~/.config/dds-monitor/alert_secret`，chmod 600；無密鑰程式無法生成有效簽章。
