# 紅隊報告 — N24：可變長度 /scan 讓整個 IDS 崩潰（Layer 3 防禦秒殺）

> 作者：紅方  日期：2026-06-02
> 攻擊者能力：L1（同 LAN 任意節點，**無密鑰**）；隔離 `ROS_DOMAIN_ID=99`
> **嚴重度：高** — 兩個封包就讓整個行為層 IDS process 死亡

---

## 一句話

攻擊者發兩個**長度不同**的 `/scan` 訊息，IDS 的 D3 偵測器用 numpy 對兩幀做逐元素運算，
**陣列長度不一致 → 未捕捉的 `ValueError` → 整個 intelligent_defense_node 崩潰退出**。
Layer 3 的全部偵測（D1–D6 + D5 心跳看門狗）瞬間歸零。

---

## 攻擊構想的演進（為什麼這次成功）

- **N19/N20（失敗）**：用「**大量小訊息**」洪水 → QoS 自動丟包擋住。
- **N24（成功）**：用「**訊息內容/形狀**」攻擊 → 不靠量，靠**畸形資料觸發程式 bug**。

這是完全不同的思路：不淹沒系統，而是**餵一個系統沒想到的輸入**。

---

## 漏洞位置

[intelligent_defense_node.py:181-199](../src/dds_security_monitor/dds_security_monitor/intelligent_defense_node.py#L181-L199) — D3 偵測器：

```python
def _detect_d3_scan_repeat(self):
    hist = list(self._scan_history)[-5:]          # 取最近 5 幀
    for i in range(1, len(hist)):
        a = np.asarray(hist[i],   dtype=np.float32)
        b = np.asarray(hist[i-1], dtype=np.float32)
        mask = np.isfinite(a) & np.isfinite(b)     # ★ a,b 長度不同 → ValueError
        ...
```

[intelligent_defense_node.py:332](../src/dds_security_monitor/dds_security_monitor/intelligent_defense_node.py#L332) — `_evaluate` 呼叫 detector **沒有 try/except**：

```python
for did, fn in [...]:
    triggered, info = fn()    # ★ 任一 detector 拋例外 → 整個 spin 崩潰
```

`_scan_cb` 收 scan 只檢查 `len(ranges) > 50` 就存進歷史，**完全不檢查「跟前一幀是否同長度」**。
→ 攻擊者送長度不一的 scan，歷史裡就混入不同長度的幀，D3 一比較就炸。

---

## 實測證據（live，IDS 真的死了）

- 環境：IDS（域 99）
- 攻擊：[N24_oversized_scan.py](N24_oversized_scan.py) 先送 500,000 點的 scan，再送 4,000,000 點的 scan

IDS 崩潰 traceback（節錄）：
```
File ".../intelligent_defense_node.py", line 191, in _detect_d3_scan_repeat
    mask = np.isfinite(a) & np.isfinite(b)
ValueError: operands could not be broadcast together with shapes (4000000,) (500000,)
[ros2run]: Process exited with failure 1   ← IDS process 死亡
```

崩潰前最後狀態：`scan_hist=20`（歷史裡混了 500k 與 4M 兩種長度的幀）→ D3 比較相鄰兩幀 → 形狀 `(4000000,)` vs `(500000,)` 無法 broadcast → 死。

---

## 關鍵：「超大」不是重點，「長度不一」才是

我用 4M 點只是剛好讓兩種長度並存。**真正的最小攻擊**根本不用超大：

> 送兩個 `/scan`，一個 `len(ranges)=100`、一個 `len(ranges)=200`（都 >50）→ 同樣崩潰。

所以這是個**極低成本、極高殺傷**的攻擊：兩個小小的、長度不同的合法型別訊息，
就讓整個 Layer 3 行為防禦下線。

---

## 影響

IDS（intelligent_defense_node）是**第三層防線**。它一死：
- **D1–D4, D6 全部停擺** → cmd_vel 劫持、scan 污染、odom 污染……全部不再被偵測。
- **D5 心跳看門狗消失** → 連「monitor 被打掛」都沒人發現了。
- 而且崩潰是**靜默的**（process 退出，沒有對外發 alert）→ 操作員不會立刻知道防線已倒。

**攻擊鏈**：先用 N24 兩個封包秒殺 IDS（無聲）→ 再用任何 cmd_vel/scan 攻擊（現在沒人看著了）。

---

## 嚴重度定位（誠實）

- **這是真崩潰**，不是「理論上會」（對比 N18 我只證實成長、沒打爆；N24 是 process 當場死亡 + 有 traceback）。
- 需要攻擊者能發 `/scan`（raw topic，預防面需 SROS2）——**但漏洞本身是純應用層的程式健壯性 bug**：
  - 就算有 SROS2，若合法 lidar 某次改變點數（換設定/換型號/動態 ROI），D3 一樣會崩 → 這也是**可靠性 bug**，不只是安全。
  - 修法完全在應用層，**不依賴 SROS2**。

---

## ⚠️ 同一個 bug 也在 D6（修補時務必一起修）

[intelligent_defense_node.py:293-295](../src/dds_security_monitor/dds_security_monitor/intelligent_defense_node.py#L293-L295) 的 D6（scan-vs-odom 一致性偵測）有**一模一樣**的寫法：
```python
a = np.asarray(hist[i],   dtype=np.float32)
b = np.asarray(hist[i-1], dtype=np.float32)
mask = np.isfinite(a) & np.isfinite(b)     # ← 長度不同照樣 ValueError 崩潰
```
差別只在 D6 要先滿足 `avg_lin > 0.05`（odom 顯示移動）才會跑到這段。
→ 攻擊者只要**搭配一筆假 /odom（速度>0.05）+ 變長度 /scan**，就算 D3 修好了，
也能走 D6 路徑把 IDS 打崩。**修補必須同時涵蓋 D3 跟 D6**（用統一的長度對齊 helper）。

> 旁註：N22（/imu 缺口）藍方**已修補** — D4 現在也監控 /imu publisher（line 229，
> 註解 "N22 修補"）。誠實更新：N22 已關閉。
> 另：odom/cmd_vel 的 NaN/Inf 輸入經 code review 為 robust（NaN 比較回 False，不崩潰），
> 此類 fuzzing 的真正命中點就是 scan 的「變長度」這條。

---

## 修補建議

1. **D3 跟 D6 比較前都先對齊長度**（最小修，做一個共用 helper）：
   ```python
   n = min(len(a), len(b))
   a, b = a[:n], b[:n]
   ```
   或直接 `if a.shape != b.shape: continue`（跳過長度不一的幀比較）。

2. **每個 detector 包 try/except**（縱深，防未來其他 detector 也有類似地雷）：
   ```python
   for did, fn in [...]:
       try:
           triggered, info = fn()
       except Exception as e:
           self.get_logger().error(f'{did} 偵測器例外，已隔離: {e}')
           triggered, info = False, ""
   ```
   讓單一 detector 的 bug 不會拖垮整個 IDS。

3. **`_scan_cb` 入口驗證**：拒絕長度異常的 scan（例如跟設定的點數差太多），
   或正規化到固定長度再存歷史。
