# 紅隊報告 — N6 / N7：操作員資訊鏈污染

> 作者：紅方  日期：2026-05-28
> 主題：**完全靜默地控制操作員看到的系統狀態**
> 攻擊者能力：**L1**（同 LAN 任意 ROS2 node）

---

## TL;DR

| 攻擊 | 目標 topic | 攻擊者偽裝 name | 結果 |
|---|---|---|---|
| **N6** | `/sensor/status` (mission_manager 的輸入) | `sensor_hub_node` (白名單) | ✗✗✗ mission_manager 被 100% 控制 |
| **N7** | `/mission/cmd` (system_status 的輸入) | `mission_manager_node` (白名單) | ✗✗✗ /system/health 顯示攻擊者字串 |

共同根因：**內部 status/mission topics 完全沒有身份驗證或簽章**，只有 `/security/alerts` 跟 `/patrol/goto` 加了 HMAC。

## 1. N6 — `/sensor/status` Spoof

### 漏洞

| 元素 | 位置 | 狀態 |
|---|---|---|
| `sensor_hub_node` 在白名單 | [config.yaml:53](../../src/dds_security_monitor/config/config.yaml#L53) | 白名單 → monitor 不 alert namesake |
| `mission_manager._on_sensor` 沒驗 publisher | [mission_manager_node.py:49-55](../../src/dds_security_monitor/dds_security_monitor/mission_manager_node.py#L49-L55) | 任何訊息看到 `'⚠️ 危險'` 字串就切換 mission |
| IDS D4 不檢查 `/sensor/status` | [intelligent_defense_node.py:173-217](../../src/dds_security_monitor/dds_security_monitor/intelligent_defense_node.py#L173-L217) | D4 只看 `/cmd_vel` 跟 `/scan` |

### 整合測試結果（22 秒）

```
attacker (名 sensor_hub_node): 每 1s 切換「危險」/「安全」字串
                                ↓ /sensor/status

mission_manager (沒驗):
   📋 任務切換 → AVOID_OBSTACLE
   📋 任務切換 → PATROL
   📋 任務切換 → AVOID_OBSTACLE
   📋 任務切換 → PATROL
   📋 任務切換 → AVOID_OBSTACLE
   📋 任務切換 → PATROL
   ... (完全跟著攻擊者腳本切換 7 輪)

monitor: (空白) — 完全沒對 sensor_hub_node 報警
```

### 實戰意義

「blinding」攻擊：真實 sensor_hub 報「⚠️ 危險」時，攻擊者以更高頻率覆蓋為「✅ 安全」
→ 操作員以為周邊無威脅，但機器人正接近障礙物 / 攻擊者正在逼近。

## 2. N7 — `/mission/cmd` Spoof（操作員面板直接覆寫）

### 漏洞

[system_status_node.py:57-58](../../src/dds_security_monitor/dds_security_monitor/system_status_node.py#L57-L58)：

```python
def _on_mission(self, msg: String) -> None:
    self._mission_status = msg.data    # ← 任意字串照單全收
```

之後 `_publish_health` 把 `_mission_status` 印在 `/system/health` 報告裡：

```python
report = (
    f'\n========== 系統健康報告 ==========\n'
    f'感測器: {self._sensor_status}\n'
    f'任務:   {self._mission_status}\n'                 # ← attacker-controlled
    f'資安:   {self._security_status}\n'
    ...
)
```

### 整合測試結果（20 秒）

攻擊者命名 `mission_manager_node`（白名單），10Hz publish 假 `/mission/cmd`。

system_status 印出的 `/system/health` 任務狀態：

```
任務:   等待任務...                               ← attacker 啟動前
任務:   等待任務...
任務:   ATTACKER_OWNED_MISSION_STATE              ← attacker 起飛
任務:   PATROL
任務:   EVERYTHING_FINE_TRUST_ME
任務:   ATTACKER_OWNED_MISSION_STATE
任務:   PATROL
任務:   EVERYTHING_FINE_TRUST_ME
任務:   ATTACKER_OWNED_MISSION_STATE              ← 攻擊持續到結束
```

monitor: 對 `mission_manager_node` namesake 沒發任何 alert。

### 對比 N6

| 維度 | N6 | N7 |
|---|---|---|
| 攻擊 topic | `/sensor/status` (input to mission_manager) | `/mission/cmd` (output of mission_manager) |
| 受害節點 | mission_manager | system_status |
| 受影響操作員界面 | 透過 mission_manager 轉發 | 直接顯示在 `/system/health` |
| 阻擋方式假設 | 修補 mission_manager 加 publisher 驗證 | 修補 system_status 加 publisher 驗證 |

**意義**：即使藍方補了 N6（mission_manager 驗 publisher），N7 還是能直達。修補必須是端到端的 — 每個訂閱者都要驗。

## 3. 完整攻擊鏈（N4 + N5 + N6 + N7）

把目前所有攻擊串起來：

```
Setup:
  1. (N5) attacker_baseline 在 monitor 啟動前進場 → 進 baseline，永久隱身

Operations:
  2. (N4) 同 process 訂閱 /security/heartbeat，forward 到 /security/alerts
         → patrol/burger_env/mission/status 全部永久 paused
  3. (N6) 偽裝 sensor_hub_node 發 /sensor/status「✅ 安全」
         → mission_manager 顯示 PATROL（即使真實感測器報危險）
  4. (N7) 偽裝 mission_manager_node 發 /mission/cmd
         → 即使藍方補了 mission_manager，operator 面板還是看到 attacker 字串

操作員看到：
  ✅ 感測器: 安全
  📋 任務:   PATROL
  ⚠️ 資安:   警報（因為 N4 持續注入 alert）

實際狀態：robot 永久停車，monitor 沒抓到任何威脅，LINE 沒收到通知
```

操作員的反應大概會是：
> 「資安系統說有警報？但感測器顯示安全、任務正常 PATROL，應該是誤報吧？」

→ 紅方達成「資安警報被操作員當噪音忽略」的最終目標。

## 4. 修補建議

### 4.1 短期（hot-fix）

對所有 status / mission / command topic 加 publisher 名稱驗證：

```python
def _on_sensor(self, msg: String) -> None:
    info = self.get_publishers_info_by_topic('/sensor/status')
    legit = [p for p in info if p.node_name == 'sensor_hub_node']
    # 如果多個同名 publisher：根據 enclave / GUID 判斷
    # 如果不認識的 publisher：直接 ignore
    ...
```

弱：只擋名字錯的，namesake 還是過得去（除非配合 SROS2 Enforce）。

### 4.2 中期：跨節點全 HMAC

把 N4 的 channel-binding 修補延伸到所有內部 topic：

```python
# sensor_hub publish:
msg.data = sign_alert(text, secret, channel="sensor/status")

# mission_manager / system_status verify:
payload = verify_alert(msg.data, secret, expected_channel="sensor/status",
                       cache=self._sensor_replay_cache)
```

每個資料流都簽，攻擊者沒 secret 就過不了。

### 4.3 根本解：SROS2 Enforce

DDS-level 限定 publisher per topic（每個 topic 定 enclave 允許清單）。
這是 threat model 2.2 已經承認的 root fix。
