# 紅隊報告 — N5：Pre-Startup Baseline Poison

> 作者：紅方  日期：2026-05-28
> 修補背景：N2 移除 `_INTERNAL_NODE_REGEX` 改用「啟動基準快照 + grace period」。
> N5 證明：**首次 poll 無條件吸收一切**，攻擊者只要比 monitor 早出現就永久隱身。

---

## TL;DR

| 漏洞 | 觸發條件 | 結果 |
|---|---|---|
| **N5** Pre-startup baseline poison | 攻擊者比 monitor 早啟動 ≥1 秒（趕在首次 poll 之前） | ✗✗✗ **成功** — monitor 從未發過 alert |

## 1. 漏洞剖析（[monitor_node.py:392-418](../../src/dds_security_monitor/dds_security_monitor/monitor_node.py#L392-L418)）

```python
def _check_graph(self) -> None:
    current: set[str] = set()
    for name, namespace in self.get_node_names_and_namespaces():
        ...
        current.add(f'{ns}/{name}')

    with self._lock:
        if not self._initialized:
            self._known_nodes = current          # ← ★ 整個 graph 都進 baseline
            self._initialized = True             #     不分白名單，不分名字
            self._init_wall_time = time.monotonic()
            self.get_logger().info(
                f'基準快照完成，記錄了 {len(current)} 個現有節點'
            )
            return                               # ← 不發 alert

        new_nodes = current - self._known_nodes
        ...
        # grace period (15s) 之後的邏輯只看 _whitelist
        grace_elapsed = time.monotonic() - self._init_wall_time
        if grace_elapsed < _STARTUP_GRACE_SEC and new_nodes:
            wl_new = {n for n in new_nodes
                      if n.rsplit('/', 1)[-1] in self._whitelist}
            if wl_new:
                self._known_nodes |= wl_new
                new_nodes -= wl_new
```

關鍵：
- N2 patch 的「grace period」設計是「對的」— 只吸收白名單的晚進場節點
- 但 grace period 之前還有一個**無條件**的 baseline snapshot
- 攻擊者卡在 monitor 啟動 → 第一次 `_check_graph` 之間（默認 5 秒）就能進 baseline

## 4. 為什麼 N2 patch 擋不住

N2 修補的設計思路：
> 「攻擊者來不及在 grace period 結束前用合法名字進場」

但這個假設假設了「monitor 比 attacker 早啟動」。實際上：
- 開發環境 / CTF：root 用 `ros2 launch` 啟動整套系統時，誰先啟動是 launch order 決定的
- 共享環境（多 user / 工廠多控制台）：attacker 可以一直 idle 在 graph 上，等 monitor 啟動
- attacker process 起動成本 < 1s，monitor 啟動到首次 poll 至少 5s（默認）

→ 攻擊者只要「比 monitor 早 1 秒」就能取得無限隱身。

## 5. 結合 N4 的全鏈攻擊

把 N5 + N4 串起來：

```
Stage 1 (N5): 攻擊者在 monitor 啟動前以非白名單名字「totally_evil_node」進場
              → 進入 baseline，永久不會被 alert
Stage 2 (N4): 同一個 attacker process 訂閱 /security/heartbeat，
              forward 到 /security/alerts
              → patrol/burger_env/mission/status 全部永久 paused
結果：       工廠/訓練系統完全停擺，monitor 從頭到尾以為一切正常
            （沒發 alert，LINE 沒有通知）
```

這是**完全靜默**的攻擊 — 操作者唯一能察覺的訊號是「patrol 一直不動」，但 alert subscriber 都顯示「收到合法警報」，看似 monitor 抓到了什麼真實威脅。實際上 monitor 啥都沒做。

## 6. 修補建議

### 選項 A（推薦）：baseline 也走 whitelist 過濾

```python
if not self._initialized:
    # 首次 poll：只把白名單上的節點吸收進 baseline
    # 非白名單者一律走 alert 邏輯
    self._known_nodes = {
        n for n in current if n.rsplit('/', 1)[-1] in self._whitelist
    }
    self._initialized = True
    self._init_wall_time = time.monotonic()
    non_wl = current - self._known_nodes
    self.get_logger().info(
        f'基準快照完成，記錄了 {len(self._known_nodes)} 個白名單節點 '
        f'(忽略 {len(non_wl)} 個非白名單者，下次 poll 會對它們 alert)'
    )
    return
```

**配套**：把這次 poll 抓到的非白名單者也直接走 `_alert_new_node`，不要等下一次 poll，否則他們又會被 `current - _known_nodes` 過濾掉。

### 選項 B：bootstrap-time 警告

啟動時印一個顯眼的 banner：
```
🚨 BASELINE SNAPSHOT：以下 N 個節點在 monitor 啟動前已存在
   - dds_security_monitor (self)
   - _ros2cli_daemon_*
   - <非白名單者> ← OPERATOR：請手動確認這個節點是合法的
```

讓 operator 在啟動時就有機會 catch poison。但這仍是「靠人類記憶」的弱保護。

### 選項 C：強制 launch order

在 launch file 中加入「等 monitor 完成首次 poll 才啟動其他節點」的依賴。
這是 ops 工程做法，不是程式碼修補。

**推薦：A + B 一起做**。
