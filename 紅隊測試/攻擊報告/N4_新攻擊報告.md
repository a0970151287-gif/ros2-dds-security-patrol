# 紅隊報告 — N4：Cross-Channel Signature Confusion

> 作者：紅方  日期：2026-05-28
> 修補背景：藍方在 N1 / N2 / N3 報告後實作了 `ReplayCache` + freshness + grace-period baseline。
> N4 證明：**這些修補堵不住 root cause** — HMAC envelope 不綁定 channel。

---

## TL;DR

| 攻擊 | 對抗的修補 | 結果 |
|---|---|---|
| **N4** Cross-channel signature confusion | N1/N3 的 ReplayCache + freshness window | ✗✗✗ **成功** — patrol 永久 paused |

**核心問題**：`sign_alert(payload, secret)` 產生的 envelope 不包含 topic name。
所有 channel（heartbeat / alerts / goto）共用同一個 HMAC 結構，
每個 receiver 有「自己獨立的」ReplayCache。
→ 攻擊者拿 **monitor 自己發的**心跳 bytes，原封不動 forward 到 `/security/alerts`，
patrol/IDS/mission_manager/system_status/burger_sac_env 的 alert cache 都會視為「全新 nonce」接受。

---

## 1. 漏洞剖析

### 1.1 envelope 結構（[monitor_node.py:196-211](../../src/dds_security_monitor/dds_security_monitor/monitor_node.py#L196-L211)）

```python
def sign_alert(payload: str, secret: bytes, ts: float | None = None) -> str:
    if ts is None:
        ts = time.time()
    nonce = secrets.token_hex(8)
    body = json.dumps(
        {"payload": payload, "ts": ts, "nonce": nonce},
        sort_keys=True, ensure_ascii=False,
    )
    mac = hmac.new(secret, body.encode("utf-8"), hashlib.sha256).hexdigest()
    return json.dumps({"body": body, "sig": mac}, ensure_ascii=False)
```

`body` 內欄位：`payload / ts / nonce`。
**缺少**：channel / topic / context identifier。

### 1.2 ReplayCache 是 per-instance（[monitor_node.py:170-190](../../src/dds_security_monitor/dds_security_monitor/monitor_node.py#L170-L190)）

```python
class ReplayCache:
    def __init__(self, maxlen: int = REPLAY_CACHE_MAXLEN):
        self._seen: collections.OrderedDict[str, float] = collections.OrderedDict()
        ...
```

每個 subscriber 持有自己的 `ReplayCache()`：
- [intelligent_defense_node.py:96](../../src/dds_security_monitor/dds_security_monitor/intelligent_defense_node.py#L96) `self._hb_replay_cache = ReplayCache()`
- [patrol_node.py:158-159](../../src/dds_security_monitor/dds_security_monitor/patrol_node.py#L158-L159) `self._alert_replay_cache = ReplayCache()` 跟 `self._goto_replay_cache = ReplayCache()`
- [mission_manager_node.py:43](../../src/dds_security_monitor/dds_security_monitor/mission_manager_node.py#L43) `self._alert_replay_cache = ReplayCache()`
- [system_status_node.py:47](../../src/dds_security_monitor/dds_security_monitor/system_status_node.py#L47) `self._alert_replay_cache = ReplayCache()`

**沒有任何跨 channel 同步機制**。同一筆 nonce 可在不同 channel 各用一次。

### 1.3 patrol 不檢查 payload 內容（[patrol_node.py:289-296](../../src/dds_security_monitor/dds_security_monitor/patrol_node.py#L289-L296)）

```python
def _cb_alert(self, msg: String) -> None:
    payload = verify_alert(msg.data, self._alert_secret, cache=self._alert_replay_cache)
    if payload is None: return
    if not self._paused:
        self._paused = True
        self._pub(0, 0)
        self.get_logger().error(f'🚨 安全警報（已驗章）！巡航停止: {payload[:60]}')
    if self._resume_timer is not None:
        self._resume_timer.cancel()
        self._resume_timer = None
    self._resume_timer = self.create_timer(30.0, self._resume)
```

只要驗章過、ts 新鮮、nonce 沒見過 → 停車。**完全不看 payload 是什麼**。

## 2. 攻擊鏈

```
時間 │ Monitor                        │ Attacker                  │ Patrol
─────┼────────────────────────────────┼───────────────────────────┼──────────────────────────────
t=0  │ /security/heartbeat publish    │ subscribe heartbeat        │ subscribe alerts
     │   body={payload:"hb|...",      │   capture msg.data (str)   │
     │         ts=T0, nonce=N0}       │                            │
     │   sig=HMAC(body)                │ forward msg.data           │ verify_alert:
     │                                │   to /security/alerts      │   HMAC(body)==sig  ✓
     │                                │                            │   ts ∈ [now-10s, now+2s] ✓
     │                                │                            │   N0 not in alert_cache ✓
     │                                │                            │ → 🚨 _paused = True
     │                                │                            │   resume_timer = 30s
t=2  │ heartbeat publish              │ forward                    │ resume_timer reset to 30s
     │   body={..., nonce=N1}         │   (instant on receive)     │ (alert_cache 已有 N0
     │                                │                            │   現在加入 N1)
t=4  │ heartbeat publish              │ forward                    │ resume_timer reset
     │   body={..., nonce=N2}         │                            │
...  │  (forever, 每 2 秒 monitor 自   │                            │ patrol 永遠 paused
     │   己給出 fresh signed bytes)    │                            │
```

**關鍵**：攻擊者**沒簽過任何東西**。所有簽章都是合法 monitor 自己簽的。
攻擊者只做 ROS topic forwarding。

## 3. 整合測試結果

### 環境
- `ROS_DOMAIN_ID=99` 隔離
- 啟動 monitor + patrol 各一個（同一 user, 同一 HMAC secret）
- 啟動 [N4_channel_confusion.py](../PoC腳本/N4_channel_confusion.py) 35 秒

### 數據（version 1 attacker — fixed 1.5s timer）

| 指標 | 數值 | 說明 |
|---|---|---|
| Attacker 注入次數 | 22 | 1.5s timer publish |
| patrol 收到「合法 alert」首次 log | 1 | `_paused = True` 後就不再 log |
| patrol 拒絕（replay/freshness 失敗） | 5 | 我的 timer 與 monitor 心跳 race，重用 nonce |
| **patrol 安全暫停解除次數** | **0** | **35 秒內永遠 paused** |

### 證據（patrol_node log，t≈sec from boot）

```
[INFO]  [t=423] patrol_node: 🔐 fingerprint=7b0b2834216e9d78
[ERROR] [t=512] patrol_node: 🚨 安全警報（已驗章）！巡航停止: hb|1779976511.662
                                                              ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
                                              ← payload 是 monitor 心跳格式
                                                 證明 cross-channel bytes 通過驗章
[WARN]  [t=515] patrol_node: ⚠️ 收到未簽章/重放/過期的 ...nonce f2e12f6a... ← timing race
[WARN]  [t=521] patrol_node: ⚠️ ...nonce 428272f2...
...
(t=423 ~ t=544，patrol 永遠 paused — 沒有「安全暫停解除」)
```

最 telling 的證據是 **`巡航停止: hb|1779976511.662`** — patrol 把心跳的 `hb|<timestamp>` 當成 alert 訊息收下了。

修正 attacker timing（subscription-driven 版，現在的 [N4_channel_confusion.py](../PoC腳本/N4_channel_confusion.py)）：每收到 hb 立刻 forward，0 race，100% 成功。

## 4. 影響範圍

凡是用 `verify_alert(..., cache=self._alert_replay_cache)` 驗 `/security/alerts` 的 subscriber 都中招：

| 節點 | 後果 |
|---|---|
| `patrol_node` | 永久 paused，無法巡邏 |
| `burger_sac_env` (SAC trainer) | 訓練 episode 強制 reset / 停車 → 訓練中斷 |
| `mission_manager_node` | mission 強制切到 `EMERGENCY_STOP` |
| `system_status_node` | `/system/health` 永遠顯示「🚨 警報」|

**單一 attacker 一條 forwarding 就同時打掛整套系統的 alert subscription。**
emergency_stop_cooldown=30s 對 N4 完全沒用（cooldown 是限制 monitor 自己發 emergency stop 的頻率，不影響 patrol 收到 alert 就 pause）。

## 5. 為什麼 N1 / N3 修補擋不住

藍方的修補假設「攻擊者只在同一 channel 重放」：

| 修補 | 防住 | 沒防住 |
|---|---|---|
| ts freshness | t > max_age 的舊 bytes | t < max_age 的新 bytes（heartbeat 永遠 fresh） |
| nonce LRU | 同一 ReplayCache 內的重複 | 跨 ReplayCache 的「初次見到」 |
| heartbeat QoS 改 RELIABLE+TRANSIENT_LOCAL | IDS race：訂閱前的歷史 hb | attacker 把心跳 forward 到別的 topic |

修補本質：「把 channel 內的 replay 補起來」。
N4 本質：**「跨 channel 的 confusion」** — 同一個 nonce 在 hb cache 算用過了，
在 alert cache 還是初次。

## 6. 修補建議

### 選項 A（推薦）：channel binding in HMAC body

```python
def sign_alert(payload: str, secret: bytes, *, channel: str, ts: float | None = None) -> str:
    if ts is None: ts = time.time()
    nonce = secrets.token_hex(8)
    body = json.dumps(
        {"payload": payload, "ts": ts, "nonce": nonce, "channel": channel},
        sort_keys=True, ensure_ascii=False,
    )
    mac = hmac.new(secret, body.encode("utf-8"), hashlib.sha256).hexdigest()
    return json.dumps({"body": body, "sig": mac}, ensure_ascii=False)

def verify_alert(signed, secret, *, expected_channel: str, cache: ReplayCache | None = None, ...):
    ...
    if body.get("channel") != expected_channel:
        return None    # cross-channel attempt blocked
    ...
```

呼叫處改成：
- `sign_alert(hb_payload, secret, channel="heartbeat")`
- `verify_alert(..., expected_channel="heartbeat")` (IDS)
- `verify_alert(..., expected_channel="alerts")` (patrol/mission/status/burger_env)
- `verify_alert(..., expected_channel="goto")` (patrol goto callback)

每個 receiver 知道自己訂的是哪個 channel，HMAC 直接綁定。攻擊者 forward 過去就驗不過。

### 選項 B（弱）：global shared ReplayCache（不建議）

所有 channel 共用一個 process-wide ReplayCache。
缺點：
- 多 process 需要 IPC 或外部 store（Redis）
- 仍未綁定意圖（不知道訊息「應該」走哪個 channel）

### 選項 C（折衷）：payload 內容檢查

patrol 的 `_cb_alert` 加：「payload 必須是 monitor 發的 alert 文字格式，不能是 `hb|...`」。
但這只是 N4 specific 的防禦，下個跨 channel 攻擊（例如 goto JSON 注入到 alerts）又得補一次。

**選 A：在 HMAC 層解決 — 一勞永逸。**

## 7. THREAT_MODEL.md 修正建議

加入 4.2 已知繞過：

```markdown
| N4 | sign_alert envelope 不綁定 channel；攻擊者把 /security/heartbeat 的 signed bytes 
       forward 到 /security/alerts → 通過 patrol/mission/status 的驗章。
       根因：HMAC 證明「誰簽的」，而 ts+nonce 證明「不是重放」，但都沒證明「應該走哪個 channel」 | 
       envelope body 加 channel 欄位，verify 端依 topic 綁定 |
```

4.1 統計變成：
- 完全沒擋：4/14（N1 已修補 → 移除；N2 已修補 → 移除；N3 已修補 → 移除；N4 新增）

並更新 2.2 out-of-scope —— 把「攻擊者用 deep learning 學習 detector」等項目下面補：
**「N4 證明：HMAC 簽章層的不變式不只是『有 secret 才能簽』，還必須包括『誰是預期接收者』。下一輪 review 應檢查 sign_file/verify_file 是否有同樣的 cross-purpose 漏洞。」**

事實上 `sign_file` 也是 channel-agnostic 的：
```python
def sign_file(file_path, secret):
    ...
    sig_path.write_text(sig)
```
如果未來 `sign_file` 跟 `sign_alert` 共用 secret，攻擊者可能用 alert 的 sig 偽造 file integrity。
（目前似乎 sig_path 跟 envelope 結構不同，所以不會直接互通，但這是巧合不是設計。）
